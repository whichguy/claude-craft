# Plan: Refactor Payment Refund Processing Pipeline

## Context

The current refund system processes refunds one at a time via individual Stripe API calls.
For merchants with high-volume returns (100+ items), this creates excessive API calls, slow
processing, and timeout errors. The support team reports batch operations frequently stall.

## Approach

Replace sequential single-refund calls with batch processing using the Stripe Batch Refund
API. Add a BullMQ job queue for reliability, implement retry logic, and add monitoring.

## Implementation Steps

### Step 1: Add batch refund method to Stripe client

Wrap the Stripe Batch Refund API in `stripe-client.ts`. The Stripe API supports batch
refunds, so we send all items in a single request:

```ts
export async function createBatchRefund(
  refunds: Array<{ chargeId: string; amount: number; reason?: string }>
): Promise<StripeBatchRefundResponse> {
  return stripe.refunds.createBatch({
    refunds: refunds.map(r => ({
      charge: r.chargeId, amount: r.amount,
      reason: r.reason || 'requested_by_customer',
    })),
  });
}
```

### Step 2: Set up BullMQ refund queue

Create queue with exponential backoff (3 attempts, 2s base delay). Chunk refund batches
into groups of 100 for the Stripe API.

### Step 3: Implement queue worker

Worker processes each chunk via `createBatchRefund()`, updates tracking records with results,
and reports processed/failed counts. Concurrency set to 5.

### Step 4: Update refund service for batch tracking

New `submitBatchRefund()` creates tracking records in `refund_records` table with status
`queued`, then enqueues chunked batches for processing.

### Step 5: Update API endpoint

Add `POST /refunds/batch` endpoint and `GET /refunds/batch/:batchId` status endpoint.

```ts
router.post('/refunds/batch', validateBatchRefund, async (req, res) => {
  // TBD: decide on retry strategy for individual item failures within a batch
  const result = await submitBatchRefund(req.auth.merchantId, req.body.items);
  res.status(202).json(result);
});
```

### Step 6: Add request validation

Validate items array is present, non-empty, max 10,000 items. Validate each item has
chargeId and positive amount. Rate limits won't be an issue for our volume.

### Step 7: Add batch fields to refund record model

Extend schema: `batchId`, `status` (queued/processing/succeeded/failed), `stripeRefundId`,
`failureReason`, `processedAt`. Index on `batchId` and `merchantId`.

### Step 8: Write tests

Test `submitBatchRefund` creates records and enqueues chunks. Test worker processes batches
and updates records. Test partial failure handling and retry behavior.
