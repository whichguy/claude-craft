# Plan: Add Email Validation to User Registration Flow

## Context

The registration endpoint accepts any string as an email address, leading to invalid
accounts, bounced welcome emails, and support tickets. Marketing reports high bounce rates
traced to invalid addresses at registration. We need format checking, domain verification,
and disposable email detection.

## Approach

Add a comprehensive `validateEmail()` function to the user service, integrate it into the
registration endpoint, add DNS MX record lookups, and maintain a disposable email blocklist.

## Files to Modify

- `src/services/user-service.ts` — Add `validateEmail()` and integrate with registration
- `src/api/routes/auth.ts` — Add validation error handling
- `src/config/email-blocklist.json` — Disposable email domain blocklist
- `src/services/user-service.test.ts` — Validation tests
- `src/api/routes/auth.test.ts` — Registration integration tests

## Implementation Steps

### Step 1: Add email validation function to user-service

In `src/services/user-service.ts`, add `validateEmail()` with: regex format check, length
validation (max 254), domain extraction, blocklist lookup against disposable providers,
and DNS MX record verification via `dns.resolveMx()`.

```ts
const EMAIL_REGEX = /^[a-zA-Z0-9.!#$%&'*+/=?^_`{|}~-]+@[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?(?:\.[a-zA-Z0-9](?:[a-zA-Z0-9-]{0,61}[a-zA-Z0-9])?)*$/;

export async function validateEmail(email: string): Promise<ValidationResult> {
  const errors: string[] = [];
  const trimmed = email.trim().toLowerCase();
  if (!EMAIL_REGEX.test(trimmed)) errors.push('Invalid email format');
  const domain = trimmed.split('@')[1];
  if (blocklist.domains.includes(domain)) errors.push('Disposable email not allowed');
  const mx = await dns.resolveMx(domain).catch(() => []);
  if (!mx.length) errors.push('Email domain does not accept mail');
  return { valid: errors.length === 0, errors };
}
```

### Step 2: Integrate into registration flow

Call `validateEmail()` at the top of `registerUser()`. Throw `ValidationError` with details
if validation fails. Then proceed to duplicate check, password hashing, and user creation.

### Step 3: Update auth route error handling

Catch `ValidationError` in `POST /register` and return 400 with error details. Catch
`ConflictError` for duplicates (409). Pass other errors to Express error handler.

### Step 4: Create disposable email blocklist

Create `src/config/email-blocklist.json` with known providers: mailinator.com,
guerrillamail.com, tempmail.com, throwaway.email, 10minutemail.com, yopmail.com.

### Step 5: Write unit tests

Test valid emails pass, empty/malformed rejected, disposable domains rejected, domains
without MX records rejected, email normalized to lowercase.

### Step 6: Write integration tests

Test `POST /register` returns 400 for invalid email, 400 for disposable, 201 for valid
registration with token in response.

## Verification

4. Manual test: registration attempts with invalid emails via Postman

## Risks

- DNS MX lookups add ~100ms latency; consider caching resolved domains
- Strict validation may reject legitimate unusual formats (plus addressing)
- Blocklist needs periodic updates as new disposable providers appear
