# Plan: Redis Cache Layer for User Profile Service

## Context

The user profile service currently hits PostgreSQL on every request. Average response
time is 180ms. We need to add a Redis cache layer to reduce latency to <20ms for
repeated reads. Profiles are read ~50x more than written.

## Current Architecture

- Express.js API server
- PostgreSQL database with `users` table
- No caching layer
- ~2000 req/s at peak

## Files to Change

| File | Change |
|------|--------|
| `src/cache/redisClient.ts` | New Redis connection wrapper |
| `src/services/userProfileService.ts` | Add cache reads before DB |
| `src/routes/profileRoutes.ts` | No changes needed |
| `config/redis.json` | Connection config (host, port, TTL) |
| `docker-compose.yml` | Add Redis container |

## Approach

Use a read-through cache pattern. On profile fetch, check Redis first. If the profile
exists in cache, return it directly. On profile update, write to DB then update cache.

## Implementation Steps

### Step 1: Redis Client Setup

Create `src/cache/redisClient.ts`:
- Initialize `ioredis` client with config from `redis.json`
- Set default TTL to 300 seconds
- Export `getCache(key)` and `setCache(key, value, ttl)` helpers
- Add connection event logging

### Step 2: Update User Profile Service

Modify `src/services/userProfileService.ts`:

```
async function getUserProfile(userId: string): Promise<UserProfile> {
  const cacheKey = `profile:${userId}`;
  const cached = await redisClient.getCache(cacheKey);

  // Return cached profile
  const profile = JSON.parse(cached);
  return profile;
}
```

- On read: check Redis with key `profile:{userId}`
- If found in cache, deserialize and return immediately
- On write: update PostgreSQL, then set cache with new value

### Step 3: Cache Invalidation on Write

Modify the `updateUserProfile` function:
- After successful DB write, call `setCache(key, updatedProfile)`
- This keeps cache consistent with DB on writes

### Step 4: Docker Compose

Add Redis 7.x container:
- Port 6379, persistent volume
- Health check with `redis-cli ping`
- Network shared with API container

### Step 5: Configuration

Create `config/redis.json`:
```json
{
  "host": "localhost",
  "port": 6379,
  "ttl": 300,
  "keyPrefix": "myapp:"
}
```

## Cache Key Strategy

- Profile: `profile:{userId}`
- Settings: `settings:{userId}`
- Keys expire after TTL (300s)

## Performance Targets

| Metric | Before | After |
|--------|--------|-------|
| Avg latency | 180ms | <20ms (cache hit) |
| P99 latency | 450ms | <50ms (cache hit) |
| DB queries/sec | 2000 | ~40 (cache misses only) |

## Verification

## Rollback

- Remove Redis container from docker-compose
- Revert userProfileService to direct DB reads
- No data migration needed (cache is ephemeral)

## Notes

- Monitor cache hit ratio via Redis INFO stats
- Consider Redis Cluster if we exceed single-node memory
- Cache warming on deploy is not needed; cache populates organically
