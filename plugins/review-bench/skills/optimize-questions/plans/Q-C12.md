# Plan: Add Retry Logic to External API Client

## Context

Our application integrates with three external services: a payment gateway, a shipping
provider, and an inventory management system. Transient failures (timeouts, 503s) cause
order processing to fail entirely. We need retry logic with exponential backoff to
improve reliability.

## Current State

- Node.js 20, TypeScript 5.3
- HTTP calls via `axios` in service-specific clients
- `src/clients/paymentClient.ts`, `src/clients/shippingClient.ts`, `src/clients/inventoryClient.ts`
- No retry logic anywhere — single attempt, throw on failure
- Existing utility: `src/utils/retry.ts` exports `withRetry(fn, options)` with:
  - Configurable max attempts, base delay, max delay
  - Exponential backoff with jitter
  - Retry-on predicate for filtering retryable errors
  - AbortSignal support for cancellation

## Approach

We will build a retry wrapper specifically designed for HTTP operations. It will
intercept axios responses, classify errors as retryable or permanent, and retry with
exponential backoff. The wrapper will be an axios interceptor so it applies automatically
to all outgoing requests.

## Files to Modify

- `src/clients/httpRetry.ts` (new) — retry interceptor for axios
- `src/clients/retryConfig.ts` (new) — per-service retry configuration
- `src/clients/paymentClient.ts` — attach retry interceptor
- `src/clients/shippingClient.ts` — attach retry interceptor
- `src/clients/inventoryClient.ts` — attach retry interceptor
- `src/types/http.ts` — retry-related type definitions

## Implementation

### Phase 1: Retry Engine

1. Create `src/types/http.ts` with retry configuration types:
   ```typescript
   interface RetryConfig {
     maxAttempts: number;
     baseDelayMs: number;
     maxDelayMs: number;
     retryableStatuses: number[];
     backoffMultiplier: number;
   }
   ```

2. Create `src/clients/httpRetry.ts` implementing retry logic:
   - `createRetryInterceptor(config: RetryConfig)` returns an axios response interceptor
   - On retryable error (timeout, 429, 500, 502, 503, 504): wait and retry
   - Delay calculation: `min(baseDelay * multiplier^attempt + jitter, maxDelay)`
   - Add `X-Retry-Attempt` header on retried requests
   - Log each retry attempt with delay and error details
   - Throw after max attempts exhausted with all attempt details

3. Create `src/clients/retryConfig.ts` with per-service defaults:
   - Payment: 3 attempts, 1s base delay (conservative — money involved)
   - Shipping: 5 attempts, 500ms base delay (more tolerant of retries)
   - Inventory: 4 attempts, 200ms base delay (fast service, quick retries)

### Phase 2: Client Integration

1. Update `paymentClient.ts`:
   - Import `createRetryInterceptor` and payment retry config
   - Attach interceptor to the axios instance

2. Update `shippingClient.ts`:
   - Same pattern, shipping-specific config
   - Add circuit breaker header on final failure

3. Update `inventoryClient.ts`:
   - Same pattern, inventory-specific config
   - Include idempotency key forwarding on retries

### Phase 3: Observability

1. Add retry metrics: `http_retry_total` counter by service and attempt number
2. Add `http_retry_exhausted` counter for fully failed retry sequences
3. Structured logging: service, attempt, delay, status code, error message

## Verification

1. Unit test retry interceptor: verify exponential backoff timing
2. Unit test: 503 on first two attempts, 200 on third — should succeed
3. Unit test: 400 error should NOT be retried (non-retryable)
4. Unit test: max attempts exhausted throws with attempt history
5. Integration test: mock server returning intermittent 503s, verify recovery
6. Verify retry headers and metrics are correctly emitted
