# Plan: Build Automated Code Review Pipeline

## Context

The team wants an automated code review system that runs on every PR. Currently reviews are
manual and inconsistent. The system should evaluate PRs against a set of quality criteria
and post findings as PR comments.

## Approach

Build a review pipeline that reads PR diffs, applies evaluation criteria, and posts results.
Use a single reviewer agent that evaluates all criteria in one pass.

## Files to Create

| File | Change |
|------|--------|
| `src/reviewer/pipeline.ts` | Main review orchestrator |
| `src/reviewer/criteria.ts` | Evaluation criteria definitions |
| `src/reviewer/commenter.ts` | GitHub PR comment posting |
| `src/reviewer/config.ts` | Per-repo review configuration |

## Implementation Steps

### Phase 1: Define Evaluation Criteria

Define the criteria battery in `criteria.ts`:

**Core criteria (load-bearing — downstream steps depend on these):**

| ID | Criterion | Spec |
|----|-----------|------|
| C1 | Correctness | Does the code do what it claims? |
| C2 | Security | Any injection, XSS, or auth bypass? |

**Derivative criteria (formatting, style — independent of core):**

| ID | Criterion | Spec |
|----|-----------|------|
| D1 | Formatting | Follow the project's prettier/eslint config. Ensure all files use consistent indentation (2 spaces for JS/TS, 4 spaces for Python). Check that import ordering follows the convention: stdlib first, then third-party packages alphabetically, then local imports with relative paths last. Verify trailing commas are used in multi-line arrays, objects, and function parameters. Confirm that line length does not exceed 100 characters (120 for comments). Check for consistent use of single quotes in JS/TS (double quotes only in JSX attributes). Ensure blank lines separate logical sections: imports, constants, class/function definitions, exports. Flag: files where >3 formatting rules are violated simultaneously. |
| D2 | Naming | Follow language conventions. Variable names should use camelCase for JS/TS and snake_case for Python. Class names must be PascalCase. Constants should be UPPER_SNAKE_CASE. Acronyms in names should follow the pattern: `httpServer` not `HTTPServer` (only first letter capitalized when mid-word). Event handler functions should be prefixed with `handle` or `on` (e.g., `handleClick`, `onSubmit`). Boolean variables should be prefixed with `is`, `has`, `should`, or `can`. Collection variables should be pluralized. Avoid single-letter variables except in lambda/arrow functions and standard loop indices (i, j, k). Hungarian notation is not allowed. Abbreviations are discouraged except for widely-known ones (e.g., `config`, `db`, `req`, `res`). |
| D3 | Documentation | Are public APIs documented? Every exported function must have a JSDoc comment with @param and @returns tags. Every exported class must have a class-level JSDoc describing its purpose and usage pattern. Every exported type/interface must have a JSDoc if its name alone doesn't fully convey its purpose. README.md must be updated if the PR adds a new module, changes configuration, or modifies the public API. CHANGELOG.md must have an entry for every user-visible change. Inline comments should explain "why" not "what" — flag comments that merely restate the code. |
| D4 | Test coverage | Are changes tested? |

### Phase 2: Build Reviewer Pipeline

1. Read PR diff from GitHub API
2. Parse changed files and extract modified functions
3. For each file, evaluate all criteria (C1, C2, D1, D2, D3, D4) in a single agent call
4. Collect findings and group by severity

### Phase 3: Build Comment Poster

1. Format findings as markdown
2. Post as PR review comment via GitHub API
3. Set PR status check based on severity (block on C1/C2, warn on D1-D4)

## Verification

1. Run on a sample PR with known issues
2. Verify all criteria produce findings
3. Check PR comment formatting
