# Plan: Payment Gateway Service

## Context

New payment gateway microservice replacing direct Stripe integration in the monolith.
Handles payment intents, webhooks, refunds, and status queries. Critical revenue path:
~2,000 payments/day, growing 20% monthly.

## Current Architecture

- Monolith with inline Stripe calls in order handlers
- No dedicated payment service
- PostgreSQL for payment records

## Files to Change

| File | Change |

| `src/server.ts` | Express server setup |
| `src/routes/paymentRoutes.ts` | Payment REST endpoints |

## Implementation Steps

### Step 1: Payments Table

`payments`: id (UUID), order_id, customer_id, amount_cents, currency, status,
stripe_payment_intent_id, stripe_charge_id, metadata (JSONB), timestamps.
Indexes on order_id, customer_id, status.

### Step 2: Stripe Client

Wrapper around Stripe SDK: `createPaymentIntent(amount, currency, metadata)` and
`refundPayment(chargeId, amount?)`.

### Step 3: Payment Service

- `initiatePayment(orderId, customerId, amountCents)`: creates Stripe PaymentIntent,
  inserts DB record with status `pending`, returns client secret.
- `getPaymentStatus(paymentId)`: queries DB by ID.

### Step 4: Webhook Handler

Verifies Stripe signature. Handles `payment_intent.succeeded` (update status, store
charge ID) and `payment_intent.payment_failed` (update status to failed).

### Step 5: REST Routes

- `POST /api/payments` — initiate payment
- `GET /api/payments/:id` — check status
- `POST /api/payments/:id/refund` — process refund

### Step 6: Kubernetes Deployment

3 replicas in `production` namespace, port 3000. Env vars from K8s Secret:
STRIPE_SECRET_KEY, DATABASE_URL.
