# Plan: Rename DB_URL Environment Variable to DATABASE_URL

## Context

Our application uses `DB_URL` as the environment variable for the PostgreSQL connection string. The industry convention is `DATABASE_URL` (used by Heroku, Railway, Render, and most ORMs by default). We need to rename this single variable across the codebase for consistency.

## Approach

Execute a coordinated multi-phase migration using a TeamCreate pattern to ensure zero downtime and backward compatibility during the transition. Each phase will be developed in parallel by dedicated agents and validated through an integration gate before merging.

## Files to Modify

- `src/db/connection.ts` — read `DATABASE_URL` instead of `DB_URL`
- `.env.example` — update variable name
- `docker-compose.yml` — update environment section
- `deploy/k8s/configmap.yml` — update config key
- `docs/setup.md` — update references
- `README.md` — update getting started section
- `src/config/env.ts` — update env validation schema
- `scripts/migrate.sh` — update env reference
- `src/db/migration-runner.ts` — update fallback env var
- `.github/workflows/ci.yml` — update env in test job
- `terraform/variables.tf` — update variable name
- `terraform/outputs.tf` — update output reference

## Implementation

### Phase 1: Discovery and Impact Analysis (Agent: impact-analyzer)

1. Spawn an impact-analyzer agent to perform a comprehensive codebase scan:
   - Search all files for `DB_URL` string literal occurrences
   - Search for `process.env.DB_URL` and `process.env['DB_URL']` patterns
   - Search documentation files for references to `DB_URL`
   - Search CI/CD configuration for `DB_URL`
   - Search infrastructure-as-code files for `DB_URL`
   - Produce an impact report listing every file and line number

2. Spawn a dependency-analyzer agent in parallel:
   - Check if any npm packages we use default to `DB_URL`
   - Check if any internal shared libraries reference `DB_URL`
   - Verify no external services push `DB_URL` to our environment
   - Cross-reference with deployment platform documentation

3. Conduct a risk assessment:
   - Classify each occurrence by risk level (runtime, build-time, docs-only)
   - Identify rollback triggers and thresholds
   - Document the blast radius for each environment (dev, staging, prod)

### Phase 2: Backward-Compatible Bridge (Agent: bridge-builder)

4. Create a compatibility bridge in `src/config/env.ts`:
   ```typescript
   const DATABASE_URL = process.env.DATABASE_URL || process.env.DB_URL;
   if (process.env.DB_URL && !process.env.DATABASE_URL) {
     console.warn('DEPRECATED: DB_URL is deprecated, use DATABASE_URL');
   }
   ```

5. Add runtime telemetry to track which variable is being used:
   - Log a structured deprecation event when `DB_URL` is detected
   - Emit a custom metric `env_var_deprecated{name="DB_URL"}` to Prometheus
   - Set up a Grafana alert if `DB_URL` usage persists beyond 30 days

6. Create a feature flag `USE_NEW_DB_URL_NAME` in LaunchDarkly:
   - Allows gradual rollout of the new variable name
   - Enables instant rollback if issues detected
   - Configure targeting rules per environment

7. Write integration tests for the bridge:
   - Test with only `DATABASE_URL` set
   - Test with only `DB_URL` set (deprecated path)
   - Test with both set (new name takes precedence)
   - Test with neither set (should throw clear error)

### Phase 3: Coordinated Replacement (Agents: code-updater, infra-updater, docs-updater)

8. Spawn code-updater agent:
   - Update `src/db/connection.ts` to import from centralized config
   - Update `src/db/migration-runner.ts` to use centralized config
   - Update `scripts/migrate.sh` to reference `DATABASE_URL`
   - Update all test files that set `process.env.DB_URL`

9. Spawn infra-updater agent in parallel:
   - Update `docker-compose.yml` environment section
   - Update `deploy/k8s/configmap.yml`
   - Update `terraform/variables.tf` variable definition
   - Update `terraform/outputs.tf` output reference
   - Update `.github/workflows/ci.yml` env block

10. Spawn docs-updater agent in parallel:
    - Update `README.md` getting started section
    - Update `docs/setup.md` configuration guide
    - Update `.env.example` with new variable name and comment
    - Search for any wiki pages referencing `DB_URL`

### Phase 4: Validation and Rollout (Agent: validation-orchestrator)

11. Run the full test suite with only `DATABASE_URL` set:
    - Unit tests: `npm test`
    - Integration tests: `npm run test:integration`
    - E2E tests: `npm run test:e2e`

12. Deploy to staging environment:
    - Set `DATABASE_URL` in staging configmap
    - Remove `DB_URL` from staging
    - Run smoke tests against staging
    - Monitor error rates for 24 hours

13. Production rollout strategy:
    - Week 1: Set both `DATABASE_URL` and `DB_URL` in production
    - Week 2: Monitor deprecation warnings in logs
    - Week 3: Remove `DB_URL` from production environment
    - Week 4: Remove the compatibility bridge code

### Phase 5: Cleanup and Verification

14. Remove backward compatibility code:
    - Delete the `DB_URL` fallback from `src/config/env.ts`
    - Remove the deprecation telemetry
    - Remove the LaunchDarkly feature flag
    - Update env validation to reject `DB_URL` with helpful error

15. Final verification:
    - Confirm no references to `DB_URL` remain in codebase
    - Confirm all environments use `DATABASE_URL`
    - Update runbooks and on-call documentation
    - Close the migration tracking ticket

## Verification

- All 3 agent outputs are reviewed and merged without conflicts
- Full CI pipeline passes in all environments
- Zero deprecation warnings in production logs after Week 3
- `grep -r "DB_URL" src/` returns zero results (excluding git history)
- Terraform plan shows no unexpected infrastructure changes
