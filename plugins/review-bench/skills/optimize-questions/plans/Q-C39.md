# Plan: Database Connection Pooling with Config-Driven Setup

## Context

Our application creates a new database connection for every query, causing connection
exhaustion under load. We need to add connection pooling with configurable pool sizes,
idle timeouts, and health checks. Configuration will be read from the application's
central config system.

## Current State

- Node.js 20, TypeScript 5.3, PostgreSQL 15
- Database connections via `pg` driver, no pooling
- Config schema (from `config/schema.ts`):
  ```typescript
  interface AppConfig {
    db: {
      hostname: string;
      port: number;
      database: string;
      credentials: { username: string; password: string };
    };
    server: { port: number; host: string };
    logging: { level: string; format: string };
  }
  ```
- Config loaded via `src/config/loader.ts` → `config.get('db.hostname')` etc.
- ~15 repository files each creating their own connections
- Production: 50+ concurrent users, PostgreSQL max_connections = 100

## Approach

Create a centralized connection pool manager that reads database configuration from
the app config and provides pooled connections to all repositories. Repositories will
request connections from the pool instead of creating their own.

## Files to Modify

- `src/database/pool.ts` (new) — connection pool manager
- `src/database/healthCheck.ts` (new) — pool health monitoring
- `src/database/types.ts` (new) — pool configuration types
- `src/repositories/baseRepository.ts` — use pool instead of direct connections
- `src/app.ts` — initialize pool on startup, shutdown on exit

## Implementation

### Phase 1: Pool Manager

1. Define pool types in `src/database/types.ts`:
   ```typescript
   interface PoolConfig {
     minConnections: number;
     maxConnections: number;
     idleTimeoutMs: number;
     connectionTimeoutMs: number;
     healthCheckIntervalMs: number;
   }
   ```

2. Create `pool.ts`:
   - `createPool(config: PoolConfig): Pool` — initialize `pg.Pool` with settings
   - Read database config from app config:
     ```typescript
     const host = config.get('database.host');
     const port = config.get('database.port');
     const dbName = config.get('database.name');
     const user = config.get('database.credentials.username');
     const password = config.get('database.credentials.password');
     ```
   - Pool settings: min 5, max 20, idle timeout 30s, connection timeout 5s
   - Connection lifecycle hooks: log connect, disconnect, error events
   - Graceful shutdown: drain pool on SIGTERM/SIGINT

3. Implement connection checkout:
   - `getConnection(): Promise<PoolClient>` — borrow from pool
   - `releaseConnection(client: PoolClient)` — return to pool
   - Auto-release wrapper: `withConnection(fn)` pattern for guaranteed release

### Phase 2: Health Monitoring

1. Create `healthCheck.ts`:
   - Periodic `SELECT 1` query every 30 seconds
   - Track pool metrics: total, idle, waiting counts
   - Emit health status events for monitoring integration
   - Mark pool as unhealthy after 3 consecutive failed checks

2. Add pool metrics endpoint:
   - `GET /health/db` returns pool status
   - Include: active connections, idle connections, wait queue size
   - Response time of last health check query

### Phase 3: Repository Migration

1. Update `baseRepository.ts`:
   - Replace `new Client()` with `pool.getConnection()`
   - Use `withConnection()` wrapper for automatic cleanup
   - Add connection error handling with retry

2. Update all 15 repository files to extend new base:
   - Remove direct `pg.Client` imports
   - Use `this.withConnection(async (client) => { ... })` pattern
   - Ensure connections are always released (finally blocks)

3. Add transaction support:
   - `withTransaction(fn)` — checkout connection, BEGIN, run fn, COMMIT/ROLLBACK
   - Nested transaction support via SAVEPOINTs

## Verification

1. Unit test pool creation: verify pg.Pool receives correct config values
2. Unit test connection checkout/release lifecycle
3. Unit test health check: healthy pool → pass, failed queries → unhealthy
4. Integration test: run 50 concurrent queries, verify pool reuse
5. Stress test: exceed max connections, verify wait queue works correctly
6. Graceful shutdown test: drain pool, verify no connection leaks
