# Plan: User Account Deactivation System

## Context

Users currently cannot deactivate their accounts. We need a deactivation flow that
anonymizes PII, revokes API keys, and marks the account as inactive. Support team
reports ~50 deactivation requests per month handled manually.

## Current State

- PostgreSQL `users` table with email, name, phone, address fields
- 12 foreign key references to `users.id` across orders, reviews, sessions, api_keys
- No soft-delete or deactivation flag exists

## Approach

Add a `deactivated_at` timestamp column and a deactivation endpoint. When triggered:
rename the `email` column to `email_hash` (stores SHA-256), backfill existing rows
with hashed values, drop the original `email` column, and update all 12 referencing
tables to handle deactivated users.

## Files to Modify

- `db/migrations/20240315_deactivate_users.sql` (new) — schema migration
- `src/services/user-service.ts` — deactivation logic
- `src/routes/users.ts` — POST /users/:id/deactivate endpoint
- `src/middleware/auth.ts` — reject auth for deactivated users
- `src/services/email-service.ts` — skip deactivated users in bulk sends

## Implementation

### Phase 1: Schema Migration

1. Add `deactivated_at` TIMESTAMP column (nullable, default null)
2. Rename `email` column to `email_hash`
3. Backfill: UPDATE users SET email_hash = SHA256(email) WHERE email IS NOT NULL
4. Drop original `email` column
5. Add index on `deactivated_at` for filtering active users

### Phase 2: Deactivation Logic

1. Create `deactivateUser(userId)` in user-service.ts:
   - Set `deactivated_at = NOW()`
   - Hash remaining PII fields (name → SHA256, phone → null, address → null)
   - Revoke all API keys for user
   - Terminate active sessions
2. Add `POST /users/:id/deactivate` endpoint with admin-only auth
3. Update auth middleware to check `deactivated_at IS NULL`

### Phase 3: Downstream Updates

1. Update email-service.ts to filter `WHERE deactivated_at IS NULL`
2. Update order-service.ts to show "[Deactivated User]" for deactivated accounts
3. Update review display to anonymize reviewer name

## Verification

1. Run migration on staging database
2. Deactivate a test user, verify PII is anonymized
3. Confirm deactivated user cannot log in
4. Verify email service skips deactivated users
5. Run full test suite
