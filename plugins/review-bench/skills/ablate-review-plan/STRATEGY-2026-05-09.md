# ablate-review-plan — Strategy Memo (2026-05-09)

## State of the harness today

Following the `micro-noclose-strict` promotion, the production `review-plan` SKILL was rebadged to be the strict variant the ablation was originally designed to test against. Concretely:

- Control = `plugins/review-suite/skills/review-plan/SKILL.md` (56 lines)
- Ablated = `plugins/review-suite/skills/review-plan/variants/SKILL-v-micro-noclose-strict.md` (34 lines)

The 22-line delta is entirely non-review metadata — full SKILL frontmatter (description, AUTOMATICALLY INVOKE triggers, model spec) and an `## ExitPlanMode Gate` section in the control. The substantive review surface (`## Directive`, `## Severity Rule (always-on)`, `## Output Format`) is **byte-identical** between the two files.

Empirically (cross-model probe-9, 2026-05-09): both control and ablated converge on `NOT_READY` under the same probe, with identical reasoning surface. The harness in its current shape is comparing two near-twins on the only axis that matters to the judge (verdict + reasoning quality), and the comparison is degenerate by construction.

Consequence: the ablation cannot answer its original question ("does the structured-question control add signal vs the directive-only ablated") because the structured-question control no longer exists in the production tree. What remains is a metadata-only delta that the judge cannot discriminate on.

## Three options

### Option 1 — Restore a structured-question control

Recreate the pre-promotion structured-question control as a third variant (e.g. `SKILL-v-q-control.md`) and run the ablation as `q-control vs micro-noclose-strict`. This restores the original ablation premise and lets the harness keep doing its job.

- **Pro:** preserves the harness's stated purpose. Re-establishes a meaningful baseline against which future variants can be judged. The historical question library (QUESTIONS.md, QUESTIONS-L3.md, QUESTIONS-UNIFIED.md) is still in-tree and can be reconstituted into a runnable variant in <1 hr.
- **Con:** maintenance overhead — the q-control needs to keep pace as questions are added/removed, otherwise it drifts back to a stale baseline. Cost ≈ a recurring small drag on every question-set change.
- **Risk:** if the q-control is restored but never updated, future ablations measure against a fossil rather than the live alternative.

### Option 2 — Retire the harness

Remove `ablate-review-plan/` entirely. Acknowledge that the question this harness was built to answer has been answered (micro-noclose-strict won; promotion happened) and the ongoing infrastructure cost is not justified by the marginal information yield.

- **Pro:** zero ongoing maintenance. Frees ~600 lines of skill code + RUN-FINDINGS bookkeeping + score-run.sh wiring. The `review-bench` umbrella has multiple other harnesses (`improve-prompt`, `compare-prompts`, `question-bench`, `validate-questions`, `optimize-questions`) that do active work.
- **Con:** loses the regression-testing capability — if someone proposes a new variant, there's no existing harness to A/B it against the current production. Would need to be rebuilt, or substituted via `compare-prompts`.
- **Risk:** the very reason the harness was built (catching regressions in the directive's review behavior) goes uncovered. A future "improvement" to `review-plan/SKILL.md` could regress without a tripwire.

### Option 3 — Pivot to cross-model-only

Keep the harness scaffold but redefine its purpose: stop comparing variants, start comparing models. Same SKILL.md run by Sonnet vs Opus vs Haiku on a fixture probe set, judging verdict-stability and inter-model agreement. The `cross-model cell` is already part of the existing pipeline — promote it to be the harness's primary output.

- **Pro:** answers a question that *is* still open (which model gives the most stable/correct review verdict on real plans) and is genuinely actionable (it informs the `model:` field of `review-plan/SKILL.md`). Reuses the existing fixture infrastructure, the judge agent, and `score-run.sh` with minor reframing.
- **Con:** changes the harness's identity — it's no longer "ablation," it's "cross-model evaluation." The SKILL frontmatter, name, and documentation all need rework. ~4 hr effort.
- **Risk:** scope creep — the same pipeline could grow to compare prompts, questions, *and* models, becoming `compare-everything` and overlapping `compare-prompts` / `compare-questions`.

## Recommendation

**Option 3 — pivot to cross-model-only.** Concrete next step:

1. Rename `ablate-review-plan` → `cross-model-review-plan` (or keep the path but rewrite the SKILL.md purpose statement to lead with cross-model evaluation; the rename can be deferred).
2. Drop the `variants/SKILL-v-micro-noclose-strict.md` invocation from the run loop. Keep the file in-tree as a historical reference; do not delete.
3. Restructure `score-run.sh` to dispatch the same `review-plan/SKILL.md` against the fixture probe set under each of {sonnet, opus, haiku}, k=3 each, and judge for verdict-agreement across models.
4. Reframe RUN-FINDINGS.md and RESULTS.md around model verdict-stability rather than variant-equivalence.
5. Defer Option 1 (restoring a q-control) until/unless someone proposes a new directive variant — at that point, the q-control becomes the comparison baseline for the variant, and the harness can run in dual mode (cross-model AND variant-comparison) for that one experiment.

Rationale for picking 3 over 2 (retire): the infrastructure exists and works; the cost of repurposing is much smaller than rebuilding from scratch when the next regression-detection need arises. Rationale for picking 3 over 1 (restore q-control): the cross-model question is the one currently producing surprising results (probe-9 cross-model NOT_READY convergence is itself worth understanding), whereas the q-control vs strict comparison's verdict has effectively been delivered by the promotion decision.

If the user disagrees with the pivot framing, Option 2 (retire) is the next-best — explicitly closing the harness is healthier than letting it drift in degenerate-comparison mode.
