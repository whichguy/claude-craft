**Flagged Issues**

- **Phase 1, Step 6 — implicit new library dependency without install path:** Step 6 edits `apps/api/src/logger.ts` to thread `traceId` via `cls-hooked` context, but no plan step installs `cls-hooked`, adds it to `package.json`, or verifies it is already present. `cls-hooked` is a native-addon-adjacent package (uses `async_hooks` and optionally native bindings); it is not a standard Node.js built-in. Treating it as available without an install step is an implicit new toolchain dependency with no build/install path. Resolution: add a step that runs `npm install cls-hooked --workspace=apps/api`, pins the version, and confirms the package builds cleanly in the CI environment.

- **Phase 1, Step 4 — registration order claim is unvalidated before the edit:** Step 3 reads `app.ts` to "confirm `authMiddleware` is currently first before route handlers," and step 4 edits `app.ts` to register `traceMiddleware` before `authMiddleware`. The correctness of the security boundary depends on `authMiddleware` remaining first among auth-relevant middleware after the insertion. If the read at step 3 reveals a different registration topology (e.g., a body-parser or CORS middleware already sits before `authMiddleware`, or `authMiddleware` is registered per-router rather than globally), the edit instruction at step 4 gives no fallback rule. This is an advisory concern: the plan should specify what to do if the pre-check reveals a topology that differs from the assumed one, rather than treating the confirmation as a formality.

- **Phase 1, Step 6 / Phase 3, Step 14 — `X-Trace-Id` header name deviates from W3C Trace Context standard without acknowledgment:** The W3C Trace Context specification (widely adopted as the interoperability standard for distributed tracing) uses `traceparent` and `tracestate` headers, not `X-Trace-Id`. The plan introduces a custom, non-standard header and treats it as the "only shared contract" without noting the deviation or explaining why the standard was not used. This is an advisory concern: any future integration with OpenTelemetry, Jaeger, Zipkin, or cloud-provider tracing will require a header migration. The plan should either adopt `traceparent` or explicitly document why a bespoke header was chosen and what the migration path looks like.

- **Phase 2, Step 10 — `crypto.randomUUID()` availability in the frontend target environment is asserted without verification:** The plan uses `crypto.randomUUID()` in the browser-side `traceClient.ts`. `crypto.randomUUID()` is available in modern browsers but not in all environments the frontend may target (e.g., older Safari, HTTP contexts where `crypto` is not exposed, Node.js test environments running jsdom). The plan provides no step to check the `browserslist` or test environment configuration, and no fallback for environments that lack the API. Resolution: add a step to verify the target browser baseline and either add a polyfill or use a UUID library already in the dependency graph.

- **Phase 3, Step 14 — test depends on `cls-hooked` context propagating correctly through Express, which is not validated in isolation:** The integration test asserts that the `traceId` appearing in the structured log matches the `X-Trace-Id` header sent by the client. This assertion is only meaningful if `cls-hooked` context is correctly bound and retrieved across the async call chain within the test server. If `cls-hooked` is misconfigured or the test server does not go through the same Express middleware stack as production (a common issue with lightweight test servers that bypass `app.ts`), the assertion will silently pass with an undefined `traceId` or fail non-deterministically. The plan should require the test server in step 14 to boot via the same `app.ts` entry point used in production, not a hand-rolled setup.

**Overall Verdict** — `NOT READY`: two blocking findings were raised (implicit new toolchain dependency `cls-hooked` with no install path; `crypto.randomUUID()` availability in the frontend target environment asserted without verification), mandating a NOT READY verdict under the severity rule.
