**Flagged Issues**

**[BLOCKING] Untrusted input crossing trust boundary without validation — Phase 1, Step 3.** The middleware reads `req.headers['x-request-id']` directly from the client and assigns the raw string to `req.requestId`, which is then passed into `logger.child({ requestId: req.requestId })` and ultimately emitted verbatim in every structured log line. No sanitization, length cap, character allowlist, or encoding step is applied. An attacker can supply a value containing newline characters (`\r\n`) or embedded JSON key-value pairs (e.g., `abc\n{"level":"error","msg":"forged line"}`), producing injected log lines or corrupted structured output downstream. Even pino's structured mode is not immune: if the transport serializes to NDJSON and the value contains a literal newline, the log stream is split into multiple JSON objects, with the injected portion appearing as a separate parseable record to log aggregators. The fix is to sanitize at the point of ingestion in `requestId.ts`: strip or reject any value containing `\r`, `\n`, or other control characters; enforce a maximum length (e.g., 128 chars); and either reject the request with a 400 or fall back to a server-generated UUID when the header is absent or invalid. The W3C Trace Context spec (`traceparent`) defines a concrete format that avoids this class of injection; adopting it here would resolve both the injection risk and the cross-service traceability design simultaneously.

**[Advisory]** The plan states the upstream nginx gateway "already injects `X-Request-Id`" and implies clients supply it too, but the verification cited is reading `src/api/server.ts:42-58` and "the request log dashboard" — neither of these is a test of gateway behavior. If the gateway ever fails to inject the header, `req.requestId` is `undefined`, and `pino.child({ requestId: undefined })` will omit the field silently rather than surfacing a missing-ID warning. The plan has no fallback: if `x-request-id` is absent, generate a UUID server-side and log a warning so gaps in header propagation are observable. This is advisory but operationally significant.

**[Advisory]** Phase 2, Steps 7–8 change the signatures of `inventory.reserve` and `payments.charge` to accept a logger as the first argument. The plan does not enumerate all existing call sites for these functions. If either is called from a background job, a queue consumer, or a test helper outside `routes/orders.ts`, those callers will break at compile time (TypeScript will catch this) but the plan gives no instruction to audit call sites before editing. The Phase 3 pre-check (`tsc --noEmit`) would surface this, but only after all edits are already made — intermediate commits after Phase 2 but before Phase 3 would be broken. Recommend adding a `grep -rn "inventory.reserve\|payments.charge"` step before Step 7 to enumerate callers.

**Overall Verdict** — NOT READY. One blocking finding: untrusted client-supplied header value flows into the log stream without sanitization, enabling CRLF/log-injection attacks at the trust boundary between external clients and the internal logging subsystem.
