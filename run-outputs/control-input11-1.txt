**Flagged Issues**

**Phase 1, Step 6 — Implicit new dependency (`cls-hooked`) without an install path.** The plan directs editing `logger.ts` to thread `traceId` via `cls-hooked` context but never adds `cls-hooked` (or its type definitions `@types/cls-hooked`) to `package.json`, nor does it verify the package is available in the monorepo workspace. `cls-hooked` relies on `async_hooks` (Node.js) and has known maintenance/compatibility caveats with newer Node versions. The plan should add a step: `npm install cls-hooked @types/cls-hooked --workspace=apps/api` and note the minimum Node version required, or replace the approach with AsyncLocalStorage (the built-in successor) which requires no extra dependency.

**Phase 1, Steps 2–4 — Non-standard header name deviates from W3C Trace Context.** The plan introduces `X-Trace-Id` as a custom header. The W3C Trace Context standard (RFC-compliant, widely adopted) specifies `traceparent` / `tracestate`. Using a bespoke `X-Trace-Id` header is reinventing ad-hoc what the ecosystem solves idiomatically, and will not interoperate with OpenTelemetry collectors, APM vendors, or any off-the-shelf distributed tracing toolchain the project may adopt later. Consider using `traceparent` per W3C Trace Context Level 1, or at minimum document explicitly why the standard header is not suitable here.

**Phase 1, Step 4 — Ordering claim is an unvalidated empirical assumption.** Step 3 reads `app.ts` to "confirm `authMiddleware` is currently first," and Step 4 registers `traceMiddleware` before `authMiddleware`. The plan asserts this ordering is correct without specifying why (e.g., auth middleware may log the request and needs the trace ID — but if so, it cannot come before the trace middleware). The correct ordering depends on whether `authMiddleware` reads `req.traceId`; the plan should state the invariant being enforced and verify it holds in the existing file.

**Phase 2, Step 10 — Each `tracedFetch` call generates a fresh UUID, breaking correlation within a single user action.** `createTracedFetch()` generates `crypto.randomUUID()` per request. If a single user action triggers multiple API calls, each gets a different trace ID, making it impossible to correlate the full action in backend logs. The plan should clarify whether per-call or per-action trace IDs are intended, and if per-action, where the ID is generated and propagated across calls.

**Phase 3, Step 14 — `traceClient` import path is unverified at integration test authorship time.** The test in `test/trace-integration.test.ts` imports `tracedFetch` from the frontend workspace (`apps/frontend/src/client/traceClient`). If the integration test lives at the monorepo root `test/` directory, it must reach across workspace boundaries. The plan does not confirm the monorepo `tsconfig` or `jest` config allows cross-workspace imports at the root level, nor does it add a path alias or note how the module resolution works. A broken import here would silently cause the test to use native `fetch` instead of `tracedFetch`, making the test invalid without failing loudly.

**Phase 3, Step 14 — Log capture mechanism is unspecified.** The test is supposed to "capture structured log output" and assert `traceId` in the log entry. The plan never specifies how structured log output is intercepted in the test environment (e.g., spy on the logger module, redirect stdout, use a test transport). Without this, the test step is underspecified and an implementer cannot write a deterministic assertion.
