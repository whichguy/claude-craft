**Flagged Issues**

- **Step 6 / `cls-hooked` dependency — implicit new toolchain without a build/install path (blocking).** Step 6 edits `apps/api/src/logger.ts` to thread `traceId` via `cls-hooked` context, but no plan step installs `cls-hooked` or its peer `async_hooks` polyfill, adds it to `package.json`, or verifies it is compatible with the Node version in use. `cls-hooked` carries known compatibility caveats with Node ≥ 16 async context propagation and is frequently superseded by `AsyncLocalStorage` (built-in since Node 12.17). The plan treats it as available, but nothing produces it. Fix: add an install step (`npm install cls-hooked --workspace=apps/api`), pin the Node version constraint, or replace with native `AsyncLocalStorage` which requires no new dependency.

- **Header naming convention — reinvention of W3C Trace Context without justification (advisory).** The plan establishes `X-Trace-Id` as the shared contract. The W3C Trace Context standard (RFC-level, widely supported) uses `traceparent` / `tracestate`. Inventing a bespoke `X-Trace-Id` header is not wrong per se, but it means existing APM agents, proxies, and structured-logging libraries that speak W3C will not auto-correlate these traces. The plan does not acknowledge this tradeoff. Consider using `traceparent` or at minimum note the deliberate divergence so future integrators are not surprised.

- **Phase 3 pre-check names Phase 1 and Phase 2 outputs but neither phase's commit is gated on the other (advisory).** Steps 8 and 13 each commit independently; there is no ordering constraint stated between them. The Phase 3 pre-check lists both outputs as required, which is correct, but because Phases 1 and 2 are described as parallel/independent the plan should explicitly state they must both be merged (or co-present on the branch) before Phase 3 begins. As written, a reader could execute Phase 2 and then jump to Phase 3 without Phase 1's commit, producing a broken intermediate test run.

- **Step 14 — `traceId` field capture mechanism unspecified (advisory).** The integration test is described as capturing "structured log output" and asserting the `traceId` field matches the `X-Trace-Id` header, but the plan does not specify how log output is intercepted (stdout capture, a log transport stub, spy on the logger module). Without a concrete interception mechanism the test step has no runnable verification path. Fix: specify the interception approach (e.g., swap the logger transport for an in-memory array in the test server setup) so the test is implementable without ambiguity.
