**Flagged Issues**

- **Phase 1, Step 6 — implicit new dependency on `cls-hooked`:** Step 6 adds `traceId` to every log entry "via `cls-hooked` context," but no prior step installs or declares this package. `cls-hooked` is an async continuation-local-storage library that requires `npm install cls-hooked` and may require Node.js `--require` setup or async_hooks compatibility configuration. The plan never adds it to `package.json`, never verifies compatibility with the monorepo's Node version, and never documents the import or initialization pattern in `logger.ts`. A senior engineer would require a step that installs the dependency and reads the current `logger.ts` initialization to confirm where CLS context is established per request — otherwise step 6 cannot be executed correctly. Fix: add an explicit install/import step and a read of the middleware registration site to determine where `cls-hooked`'s namespace `run()` call belongs.

- **Phase 1, Step 4 — ordering claim is an empirical assertion not verified before edit:** Step 4 edits `app.ts` to register `traceMiddleware` *before* `authMiddleware`, but step 3 only says to "confirm `authMiddleware` is currently first before route handlers." The reasoning for placing `traceMiddleware` ahead of `authMiddleware` is not examined: if auth middleware short-circuits unauthenticated requests before `traceMiddleware` runs, trace IDs will be absent from rejection logs — the opposite of the stated goal. Step 3 should also check whether placing trace *before* auth causes any ordering side effects (e.g., auth middleware that reads a request attribute set by trace). Fix: explicitly verify that tracing before auth is safe and document the rationale, or reverse the order and explain why auth-first is still acceptable for trace logging.

- **Phase 1, Step 6 — `req.traceId` type augmentation is missing:** `traceMiddleware.ts` sets `req.traceId` on the Express `Request` object, but TypeScript will error on this unless `express-serve-static-core`'s `Request` interface is augmented. No plan step creates or modifies a type declaration file (`types/express/index.d.ts` or similar). This will surface as a type error when step 16 runs `tsc --noEmit`, making the integration phase fragile. Fix: add a step in Phase 1 to create or update an Express type augmentation file and verify it is picked up by `tsconfig.json`.

- **Phase 2, Step 10 — `crypto.randomUUID()` availability assumption:** The plan generates trace IDs in the frontend with `crypto.randomUUID()`. This is a Web Crypto API method not available in all browser environments the frontend may target (e.g., non-HTTPS contexts, older browsers). The plan makes no reference to a polyfill, a build target check, or an existing UUID utility already present in the frontend codebase. Fix: verify `crypto.randomUUID()` is acceptable for the declared browser targets or document that the frontend is HTTPS-only; alternatively, adopt the same UUID library used elsewhere in the monorepo.

- **Phase 3, Step 14 — test captures structured log output with no capture mechanism specified:** Step 14 asserts "`traceId` in log entry matches `X-Trace-Id` value sent by the client" by capturing "structured log output," but no step specifies how the test intercepts logger output. The logger writes to some output sink (stdout, file, or transport); without a logger spy/mock or a way to capture in-process output, the assertion is unrunnable. The plan does not indicate whether the logger can be switched to a testable transport in test context. Fix: add a step that reads the logger's test configuration or describes how to inject a capture transport before the assertion is written.

- **Advisory — header name `X-Trace-Id` deviates from W3C Trace Context standard:** The plan invents a custom header `X-Trace-Id` with a UUID format when the W3C `traceparent` / `tracestate` headers (RFC-standardized) exist precisely for this purpose and are natively supported by OpenTelemetry and most observability backends. The plan has no explanation of why a bespoke header is preferred. This is not blocking, but a senior engineer would at minimum note the tradeoff. Consider adopting `traceparent` to avoid reimplementing a wheel and to ease future integration with distributed tracing backends.
