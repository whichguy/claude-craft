**Flagged Issues**

- **Phase 1, Step 3 — Untrusted input crossing trust boundary without sanitization:** The middleware reads `req.headers['x-request-id']` directly from the client and sets `req.requestId = headerValue` without any validation, sanitization, or length cap. This value is then passed to `pino`'s child logger as `{ requestId: req.requestId }`. Pino serializes structured fields as JSON, so bare CRLF sequences in the header value (`\r\n`) will be embedded in the structured log output as escaped characters in JSON mode — however, if a log shipper or downstream consumer processes the raw string value (e.g., a text-mode transport, a grep pipeline, a Splunk/ELK field extractor using raw string parsing, or a log dashboard that renders the field value), an attacker-controlled header can inject forged log lines or corrupt log structure. The plan's stated rationale explicitly says clients supply this header and the gateway propagates it, making the value user-controlled by definition. The fix is to validate the incoming value against a safe pattern (e.g., `/^[a-zA-Z0-9\-]{1,128}$/`) and either reject/strip non-conforming values or fall back to a server-generated UUID. This is the classic untrusted-input-to-trust-boundary issue: the plan must add a validation/sanitization step inside `requestIdMiddleware` before assigning to `req.requestId`.

- **Phase 1, Step 3 — No fallback when header is absent:** The middleware reads `req.headers['x-request-id']` and assigns it directly. If the header is missing, `req.requestId` is `undefined`, and `logger.child({ requestId: undefined })` will either omit the field silently or serialize as `"requestId":null`/`"requestId":undefined` depending on pino version — breaking the guarantee that every log line carries a correlation ID. The plan does not specify the fallback behavior (generate a UUID server-side if the header is absent). This must be addressed in Step 3.

- **Phase 2, Step 9 — Test only covers the happy path:** The unit test asserts `req.requestId === 'abc123'` for a well-formed header, but there is no test for a missing header (fallback behavior), an oversized header, or a header containing CRLF/special characters. Without these cases the validation/sanitization logic added to fix the above finding has no coverage. The plan should add negative test cases to `test/middleware/requestId.test.ts`.

- **Phase 2, Pre-check — "manual curl test" is not a runnable verification command:** The Phase 2 pre-check says "req.log is populated on a sample request (manual curl test)" but gives no actual command. A concrete `curl -s -H "X-Request-Id: test-1" http://localhost:3000/orders | ...` invocation should be specified, or this should be folded into the integration test rather than left as an informal manual step.
