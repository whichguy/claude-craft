**Flagged Issues**

- **Phase 1, Step 6 — unverified `cls-hooked` dependency:** The plan introduces `cls-hooked` (a continuation-local storage library) to thread `traceId` through the logger without any step that installs or declares it. There is no `npm install cls-hooked --workspace=apps/api`, no `package.json` edit, and no verification that the monorepo's TypeScript build config includes types for it. `cls-hooked` also carries a known risk on Node ≥ 12 with async_hooks stability; the plan presents it as a settled choice without noting the alternative of passing `traceId` explicitly through a request-scoped logger. The fix: add an installation step, cite why `cls-hooked` was chosen over explicit parameter threading, and confirm `@types/cls-hooked` exists or is installed.

- **Phase 1, Step 4 — registration order assumption without pre-read of full middleware chain:** Step 3 reads `app.ts` to confirm `authMiddleware` is currently first, but Step 4 registers `traceMiddleware` *before* `authMiddleware` without stating why ordering relative to any other middleware (e.g., body-parser, cors, error handlers) is correct. If `authMiddleware` itself reads `req.traceId` for logging, placing `traceMiddleware` first is correct — but if there are other middleware between the top and `authMiddleware`, a naive prepend may interleave incorrectly. The fix: the step should explicitly state "insert at position 0" or "insert immediately before `authMiddleware`" based on what Step 3 observes, not just "register before authMiddleware".

- **Phase 2, Step 10 — `crypto.randomUUID()` availability in browser context not verified:** The plan uses `crypto.randomUUID()` in the frontend client. This API is available in modern browsers but requires a secure context (HTTPS or localhost). The plan makes no mention of a polyfill or feature-detection for older browsers or non-secure contexts. If the app targets environments without `crypto.randomUUID()`, every traced request will throw at runtime. The fix: add a note confirming the minimum browser target supports `crypto.randomUUID()`, or add a fallback (e.g., `uuid` npm package).

- **Phase 1/2 — header name `X-Trace-Id` diverges from W3C Trace Context standard:** The plan invents a bespoke `X-Trace-Id` header rather than using the W3C `traceparent`/`tracestate` headers (RFC-standard, supported by OpenTelemetry, Datadog, Jaeger, etc.). This is an advisory concern: the plan will work as written, but it locks the team out of drop-in OTel integration later and requires custom parsing on any third-party ingestion side. The fix: consider using `traceparent` per W3C Trace Context Level 1 (widely supported, no additional deps); if `X-Trace-Id` is a deliberate constraint (e.g., legacy infrastructure), add a comment stating why.

- **Phase 3, Step 14 — test captures log output but no mechanism specified:** The integration test asserts "`traceId` in log entry matches `X-Trace-Id` value sent by the client" but does not specify how structured log output is captured during the test. If `logger.ts` writes to stdout or a file, the test needs to intercept or redirect that output. The plan provides no step to configure a test-mode logger sink or spy on the logger module. Without this, the assertion has no target to read, and the test will either always pass vacuously or fail to compile. The fix: add a step that either stubs the logger to collect entries in memory for the test, or configures a test transport that captures structured output.

- **Phase 3, Step 16 — `tsc --noEmit` scope ambiguity:** The plan says "zero type errors across both workspaces" but `tsc --noEmit` with no path argument will use whichever `tsconfig.json` is nearest in the working directory. In a monorepo with per-workspace tsconfigs, this may only check one workspace. The fix: run `tsc --noEmit` explicitly per workspace (e.g., `tsc --noEmit -p apps/api/tsconfig.json && tsc --noEmit -p apps/frontend/tsconfig.json`) or use a root composite tsconfig that references both.
