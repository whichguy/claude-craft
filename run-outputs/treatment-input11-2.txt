**Flagged Issues**

- **Step 6 — `cls-hooked` introduced without build/install path:** The plan edits `apps/api/src/logger.ts` to add `traceId` via `cls-hooked` context, but `cls-hooked` is an external npm package that is not mentioned anywhere in the plan as a dependency to install. There is no `npm install cls-hooked --workspace=apps/api` step, no package.json update, and no verification that it is already present. This is an implicit new runtime dependency without a build/install path. The fix is to add an explicit install step before Step 6 and confirm the package is compatible with the Node version in use.

- **Step 6 — `cls-hooked` usage unvalidated against existing logger pattern:** The plan reads `apps/api/src/logger.ts` at Step 5 to verify the current log format, but does not verify whether the logger is already using a context-propagation library or has any async-context setup. `cls-hooked` requires that the CLS namespace be created and bound before any async operations; if the logger is used in middleware that runs before CLS context is established, `traceId` will be `undefined` in log entries. There is no verification step or runnable test assertion that confirms the CLS context is actually populated at log time in all code paths (e.g., async route handlers, error handlers, background tasks). The plan should add a runnable assertion — e.g., a unit test that confirms `traceId` is present in a log entry from inside an async handler — before the Phase 1 commit.

- **Phase 2, Step 10 — `crypto.randomUUID()` availability in browser context unverified:** `traceClient.ts` generates a fresh `crypto.randomUUID()` per request in the frontend. `crypto.randomUUID()` is not available in all browser contexts targeted by the project (notably not in non-secure origins / HTTP, and not in older Safari versions before 15.4). The plan makes no mention of browser compat requirements or a polyfill, and no step checks the frontend's `tsconfig.json` `lib` target or browser support matrix. This is an unvalidated empirical assumption (that `crypto.randomUUID()` is universally available in the frontend environment). The fix is to read `apps/frontend/tsconfig.json` and the project's browser target config, and either confirm `crypto.randomUUID()` is available or add a fallback UUID generator.

- **Phase 3, Step 14 — Integration test captures "structured log output" with no mechanism specified:** The test plan says "Capture structured log output — assert `traceId` in log entry matches `X-Trace-Id` value sent by the client," but no step establishes how the test server's log output is intercepted. If `apps/api/src/logger.ts` writes to stdout/file, a test runner would need to spy on the logger or redirect its transport. There is no read of the logger's existing transport configuration, no mock/spy setup described, and no runnable command that would fail if the capture mechanism is missing. This is a verification step with no runnable mechanism. The fix is to specify the logger interception strategy (e.g., spy on the logger's `info` method, or configure a test transport) before Step 14 is written.

- **Advisory — `X-Trace-Id` header deviates from W3C Trace Context standard (`traceparent`):** The plan establishes a custom `X-Trace-Id` / UUID-only contract rather than the W3C Trace Context `traceparent` header format (RFC spec: `00-<trace-id>-<parent-id>-<flags>`). Any future integration with OpenTelemetry collectors, Datadog, or other distributed tracing backends will require re-mapping this custom header. Consider adopting `traceparent` now, or at minimum document why the W3C standard was rejected.

- **Advisory — Per-request `crypto.randomUUID()` in `traceClient.ts` generates a new trace ID per fetch call rather than per user session or logical operation:** Distributed tracing typically propagates a single trace ID across all spans of one logical operation, not a new ID per HTTP call. The plan's design means that a multi-call sequence (e.g., auth + data fetch + analytics) produces uncorrelated trace IDs in the backend logs, defeating the stated goal of correlating a frontend request failure with its backend log entry when multiple calls are in flight. This is a design-level advisory; if per-call IDs are intentional, the plan should document why.

**Overall Verdict** — NOT READY: two or more blocking findings are present — an implicit new runtime dependency (`cls-hooked`) without a build/install path, and an unvalidated browser-compat assumption for `crypto.randomUUID()` in the frontend — each independently triggering the blocking threshold.
