**Flagged Issues**

**Phase 1, Step 3 — Untrusted input crossing trust boundary without sanitization (blocking).** The middleware reads `req.headers['x-request-id']` and passes the raw string directly into `req.requestId` and then into `logger.child({ requestId: req.requestId })`. The header value is fully attacker-controlled. While pino serializes structured fields as JSON (reducing CRLF risk in pino's own output), the plan calls for passing `req.requestId` as a plain string field with no validation or sanitization. If any downstream code concatenates `req.requestId` into a string log message, or if a non-pino transport (e.g., a text formatter, a legacy `console.log`, or a future log shipper) emits the field as plain text, a value like `abc\nINFO fake-event: ...` produces forged log lines. The plan presents no validation step. Fix: add a validation/sanitization step in the middleware — e.g., strip or reject values containing `\r`, `\n`, or control characters; optionally enforce UUID format or a max length. Cite the step where this happens and what the enforcement rule is.

**Phase 1, Step 3 — No fallback when header is absent.** The plan sets `req.requestId = headerValue` where `headerValue` may be `undefined` if a client omits the header. The subsequent `logger.child({ requestId: undefined })` will serialize as `"requestId":null` or omit the field depending on pino version, which may cause downstream filtering/alerting to behave unexpectedly. The plan asserts the gateway always injects the header (citing `infra/gateway/nginx.conf:88`) but does not verify whether direct-to-service calls (internal tooling, health checks, tests) bypass the gateway. Fix: generate a UUID fallback (`req.requestId = headerValue ?? uuid()`) and add a note on whether server-generated IDs are acceptable for non-gateway paths.

**Phase 2, Step 6 — Unverified line references.** The plan cites `lines 24, 47, 89` in `routes/orders.ts:placeOrder` as the exact locations of `logger.info` calls to replace. No pre-read of `routes/orders.ts` is listed in Phase 2's pre-checks. If those line numbers have drifted (or were assumed), the implementer will either miss calls or edit the wrong lines. Fix: add a pre-read of `routes/orders.ts` (and the service files) in Phase 2's pre-check block, confirm line numbers match, and remove the hardcoded line citations in favor of "all `logger.*` calls in `placeOrder`."

**Phase 2, Pre-check — "manual curl test" is not a runnable command.** The Phase 2 pre-check says `req.log` is populated "on a sample request (manual curl test)" but gives no command. This leaves the verification step non-reproducible and easy to skip. Fix: include the exact curl command (e.g., `curl -s -H "X-Request-Id: smoke-1" http://localhost:3000/orders`) and specify what log output is expected to confirm `req.log` is wired.

**Phase 3, Step 9 — Unit test does not cover the missing-header path.** The described test only covers the happy path (`X-Request-Id: abc123`). Given the missing-fallback issue above, the absence of a test for a request with no header means a regression to `undefined`/`null` propagation would not be caught. Fix: add a test case for absent header asserting that `req.requestId` is a non-empty string (UUID or other generated value).
