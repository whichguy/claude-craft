# Plan: Add Comment System to Blog Platform

## Context

The blog platform currently has no user interaction. Posts are read-only. Product wants
threaded comments with markdown, nested replies (up to 3 levels), author attribution, and
moderation. The platform uses Express + PostgreSQL + React.

## Approach

Build the full comment system: database schema, API endpoints, React components, and
moderation tools. Use materialized path pattern for efficient tree queries.

## Files to Modify

- `migrations/20240315_create_comments.sql` — Schema with materialized path
- `src/models/comment.ts` — Data model and query functions
- `src/api/routes/comments.ts` — REST endpoints (CRUD + flag + moderate)
- `src/components/CommentSection/CommentSection.tsx` — Main container
- `src/components/CommentSection/CommentItem.tsx` — Individual comment with replies
- `src/components/CommentSection/CommentForm.tsx` — Comment input with markdown preview
- `src/pages/BlogPost/BlogPost.tsx` — Integrate comment section
- `src/admin/components/ModerationQueue.tsx` — Admin moderation interface
- `tests/comments.test.ts` — All tests

## Implementation Steps

### Step 1: Create database migration

Create `comments` table with: id (UUID), post_id (FK), author_id (FK), parent_id (self-FK),
materialized_path, depth (check <= 3), body, body_html, status (pending/approved/rejected/
flagged), flag_count, timestamps. Add indexes on post_id, parent_id, materialized_path,
status, author_id.

### Step 2: Implement comment data model

`createComment()`: render markdown to HTML, compute materialized path from parent, enforce
depth limit. `getCommentsForPost()`: join with users, filter approved, order by path then
date, build tree structure. `moderateComment()`: update status. `flagComment()`: increment
count, auto-flag at threshold of 3.

### Step 3: Create API endpoints

- `GET /posts/:postId/comments` — Threaded tree for a post
- `POST /posts/:postId/comments` — Create comment (auth required, accepts parentId)
- `POST /comments/:commentId/flag` — Flag a comment (auth required)
- `PATCH /admin/comments/:commentId` — Moderate: approve/reject (admin required)

### Step 4: Build CommentSection component

Fetches comments on mount, renders `CommentForm` for new top-level comments, maps comment
tree to `CommentItem` components. Handles optimistic insertion on new comment.

### Step 5: Build CommentItem with nested replies

Shows author avatar/name, relative timestamp, sanitized HTML body, reply button (hidden at
depth 3), flag button. Inline reply form toggle. Recursive child rendering with indent.

### Step 6: Build CommentForm and admin ModerationQueue

CommentForm: markdown preview toggle, character count, submit with loading state.
ModerationQueue: filterable table of pending/flagged comments, bulk approve/reject, preview.

### Step 7: Integrate into BlogPost page

Add `<CommentSection postId={post.id} />` below the post content in `BlogPost.tsx`.

### Step 8: Write tests

Model tests: create top-level, create nested reply with correct path, reject depth > 3,
markdown rendering. Query tests: threaded tree, exclude non-approved, ordering. API tests:
201 for valid, 401 unauth, 400 empty body. Moderation: approve, reject, admin-only. Flag:
increment count, auto-flag at 3.

## Verification

1. Run migration on dev database
2. `npm test` — all tests pass
3. `npx tsc --noEmit` — type checking
4. Manual: post comment, reply, verify threading
5. Manual: flag comment, verify auto-flag threshold
6. Manual: admin moderation approve/reject
7. Load test: 1000 comments on single post

## Risks

- Markdown rendering must sanitize HTML to prevent XSS (use DOMPurify)
- Materialized path can be slow for very deep re-ordering
- Comment email notifications are out of scope
