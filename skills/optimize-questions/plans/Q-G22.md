# Plan: Plugin Architecture for Data Processors

## Context

Our ETL pipeline (`src/pipeline/`) currently has all data processors hardcoded in a switch statement. Adding a new processor requires modifying core pipeline code, recompiling, and redeploying. We need a plugin architecture where processors are loaded dynamically from a `plugins/` directory and registered via a manifest file.

## Approach

Introduce a plugin manifest system where each plugin declares its capabilities in a `plugin.json` file. A plugin loader reads these manifests at startup, validates the declared interfaces, and registers processors with the pipeline. This decouples processor development from the core pipeline.

## Files to Modify

- `src/pipeline/plugin-loader.ts` — new dynamic plugin discovery and loading
- `src/pipeline/registry.ts` — new plugin registry with lifecycle hooks
- `src/pipeline/engine.ts` — update to use registry instead of switch
- `src/pipeline/config-reader.ts` — new pipeline configuration reader
- `plugins/csv-processor/index.ts` — example CSV plugin
- `plugins/csv-processor/plugin.json` — example manifest
- `types/plugin.d.ts` — plugin interface definitions
- `package.json` — add `glob` and `ajv` dependencies

## Implementation

### Phase 1: Plugin Interface and Discovery

1. Install dependencies:
   ```
   npm install glob ajv fast-glob
   ```

2. Define the plugin interface in `types/plugin.d.ts`:
   ```typescript
   interface PluginManifest {
     name: string;
     version: string;
     processors: string[];
     config_schema?: object;
   }

   interface DataProcessor {
     name: string;
     process(input: Buffer, options: Record<string, unknown>): Promise<Buffer>;
     validate?(input: Buffer): Promise<boolean>;
   }
   ```

3. Create `src/pipeline/plugin-loader.ts`:
   - Scan `plugins/*/plugin.json` using fast-glob
   - Parse and validate each manifest against a JSON schema
   - Dynamically `import()` each plugin's entry point
   - Return an array of loaded plugin modules

4. Create `src/pipeline/registry.ts`:
   - `register(processor: DataProcessor)` — adds to internal map
   - `get(name: string)` — retrieves processor by name
   - `list()` — returns all registered processor names
   - `initialize()` — calls `onInit()` lifecycle hook on each plugin

### Phase 2: Pipeline Configuration

5. Create `src/pipeline/config-reader.ts`:
   - Read `pipeline.config.json` from the project root
   - This file specifies which plugins to enable and their settings:
     ```json
     {
       "plugins": {
         "csv-processor": { "delimiter": ",", "encoding": "utf-8" },
         "json-processor": { "pretty": true }
       },
       "pipeline": {
         "concurrency": 4,
         "timeout_ms": 30000
       }
     }
     ```
   - Validate plugin config against each plugin's declared `config_schema`
   - Merge defaults with user-provided overrides

6. Update `src/pipeline/engine.ts`:
   - Remove the hardcoded processor switch statement
   - Accept a registry instance in the constructor
   - Resolve processor by name via `registry.get(name)`
   - Pass validated config to each processor's `process()` method

7. Update startup in `src/server.ts`:
   - Call `pluginLoader.discover()` to find all plugins
   - Call `registry.initialize()` with discovered plugins
   - Pass the populated registry to the pipeline engine

### Phase 3: Example Plugin and Validation

8. Create `plugins/csv-processor/plugin.json`:
   ```json
   {
     "name": "csv-processor",
     "version": "1.0.0",
     "entry": "./index.ts",
     "processors": ["csv-parse", "csv-transform"],
     "config_schema": {
       "type": "object",
       "properties": {
         "delimiter": { "type": "string", "default": "," },
         "encoding": { "type": "string", "default": "utf-8" }
       }
     }
   }
   ```

9. Implement `plugins/csv-processor/index.ts`:
   - Export `csvParse` and `csvTransform` processors
   - Each implements the `DataProcessor` interface
   - Use `papaparse` for CSV parsing with streaming support

10. Create `plugins/json-processor/plugin.json` and `index.ts`:
    - JSON-to-JSON transformation processor
    - Supports JSONPath-based field extraction
    - Exports `transformFields` utility for use by other plugins

### Phase 4: Pipeline Orchestration

11. Add pipeline composition in `src/pipeline/orchestrator.ts`:
    - Read pipeline stage definitions from `pipeline.config.json`
    - Each stage references a processor by name and passes config
    - Stages execute sequentially, output of one feeds input of next
    - Call `registry.get(stageName).process()` for each stage

12. Implement error handling in the orchestrator:
    - If a processor throws, catch and log the error with stage context
    - Support `on_error: skip | abort | retry` per stage
    - Dead-letter queue for failed records

13. Add `src/pipeline/metrics.ts`:
    - Track records processed, failures, and duration per stage
    - Expose via `/metrics` endpoint for Prometheus scraping
