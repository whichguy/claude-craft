# Plan: E-Commerce Product Catalog — Database + API + Admin UI

## Context

We're building a product catalog system for our e-commerce platform. Products have
variants (size, color), belong to categories with a tree hierarchy, and need full-text
search. The current system uses a flat JSON file that's manually edited — it can't
handle the 5,000+ products we're scaling to.

## Current State

- Products stored in `data/products.json` (flat file, 800 products)
- No variants, no category hierarchy, no search
- Admin edits JSON directly, deploys to update
- Frontend fetches entire JSON on page load

## Approach

Migrate to PostgreSQL with a proper schema supporting variants and category trees.
Build a REST API for CRUD operations and full-text search. Add an admin UI for
product management. Three-phase rollout: database first, then API, then admin UI.

## Files to Modify

- `db/migrations/001_products.sql` (new) — product schema
- `db/migrations/002_categories.sql` (new) — category tree schema

- `src/api/products.ts` (new) — product CRUD endpoints

## Implementation

### Phase 1: Database Schema & Migration

This phase establishes the relational foundation for the product catalog.
We prioritize getting the schema right because the API and admin UI both depend on it,
and schema changes after data is loaded are expensive.

1. Create `001_products.sql`:
   - `products` table: `id`, `name`, `slug`, `description`, `base_price`, `status`,
     `created_at`, `updated_at`
   - Add `tsvector` column for full-text search, with GIN index
   - Add trigger to auto-update tsvector on insert/update
2. Create `002_categories.sql`:
   - `categories` table: `id`, `name`, `slug`, `parent_id` (self-referential)
   - `product_categories` junction table for many-to-many
   - Add ltree extension for efficient tree queries
3. Create `003_variants.sql`:
   - `variants` table: `id`, `product_id`, `sku`, `price_modifier`, `stock_count`
   - `variant_options` table: `variant_id`, `option_name`, `option_value`
   - Foreign key from variants to products with CASCADE delete
4. Create `import.ts` to read `products.json` and insert into new schema
5. Run migration on staging, verify import produces correct row counts

### Phase 2: Database Migration

1. Create `GET /api/products` — list with pagination, filtering, sorting
2. Support query params: `?page=1&limit=20&sort=price&category=shoes`
3. Create `GET /api/products/:slug` — single product with variants
4. Create `POST /api/products` — create product (admin auth required)
5. Create `PUT /api/products/:id` — update product
6. Create `DELETE /api/products/:id` — soft delete (set status=archived)
7. Create `GET /api/categories` — full tree structure
8. Create `POST /api/categories` — add category with parent_id
9. Create `GET /api/search?q=term` — full-text search using tsvector
10. Add input validation with `zod` schemas for all endpoints
11. Add pagination metadata in response: `{ data, total, page, pages }`

### Phase 3: API Endpoints

1. Create `ProductForm.tsx` with fields for all product attributes
2. Add variant management: add/remove/edit variants inline
3. Add image upload with drag-and-drop (store in S3, save URL in DB)
4. Create `CategoryTree.tsx` with expandable tree view
5. Support drag-and-drop category reordering
6. Create `VariantTable.tsx` for bulk variant editing
7. Add auto-generated SKU suggestions based on product name + options
8. Wire forms to API endpoints with optimistic updates
9. Add search bar that hits `GET /api/search` with debounced input
10. Add bulk actions: archive multiple products, move to category

## Verification

1. Migration tests: run up/down migrations, verify schema matches spec
2. Import test: import fixture JSON, verify row counts and relationships
3. API tests: CRUD operations for products, categories, variants
4. Search test: insert products, search by term, verify relevance ordering
5. Admin UI: Cypress tests for product creation and category management
6. Performance: query products with 5,000 rows, verify <200ms response
7. Verify cascade delete removes variants when product is deleted
