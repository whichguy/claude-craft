# Plan: Payment Gateway Integration

## Context

Our e-commerce platform needs to integrate with a new payment gateway (Adyen) alongside
the existing Stripe integration. This enables geographic expansion into markets where
Stripe is unavailable. The integration must support credit cards, bank transfers, and
local payment methods (iDEAL, SEPA, Boleto).

## Current State

- Node.js 20, TypeScript 5.3, Express 4.18
- Existing Stripe integration in `src/payments/stripeProvider.ts`
- Payment interface: `PaymentProvider` with `charge()`, `refund()`, `getStatus()`
- Order flow: cart → checkout → payment → confirmation
- Monthly transaction volume: ~15,000 orders, $2.1M GMV
- PCI DSS Level 2 compliance (SAQ A-EP)

## Approach

Build an Adyen payment provider that implements the existing `PaymentProvider` interface.
Add a router that selects the appropriate provider based on customer region and payment
method. Implement the three supported payment methods with Adyen-specific flows.

## Files to Modify

- `src/payments/adyenProvider.ts` (new) — Adyen implementation of PaymentProvider
- `src/payments/adyenClient.ts` (new) — low-level Adyen API client
- `src/payments/paymentRouter.ts` (new) — provider selection logic
- `src/payments/methods/ideal.ts` (new) — iDEAL redirect flow handler
- `src/payments/methods/sepa.ts` (new) — SEPA direct debit handler
- `src/payments/methods/boleto.ts` (new) — Boleto voucher generation handler
- `src/types/payment.ts` — extend PaymentMethod enum with new methods
- `src/config/adyenConfig.ts` (new) — Adyen API credentials and merchant config
- `src/routes/payments.ts` — add Adyen webhook endpoint

## Implementation

### Step 1: Adyen API Client

1. Create `adyenConfig.ts`:
   - API key, merchant account, environment (test/live) from env vars
   - Endpoint URLs for checkout, payments, and modifications APIs
   - HMAC key for webhook verification

2. Create `adyenClient.ts`:
   - HTTP client wrapper for Adyen REST API
   - Request signing with API key
   - Response parsing and error mapping
   - Retry logic for transient failures (503, timeouts)

### Step 2: Payment Provider

1. Create `adyenProvider.ts` implementing `PaymentProvider`:
   - `charge(amount, currency, paymentMethod, metadata)`:
     - Create payment session via `/payments` endpoint
     - Handle 3DS redirect if required
     - Return payment reference for tracking
   - `refund(paymentRef, amount)`:
     - Submit refund via `/payments/{ref}/refunds`
     - Support partial and full refunds
   - `getStatus(paymentRef)`:
     - Query payment status via `/payments/{ref}/details`
     - Map Adyen statuses to our internal status enum

### Step 3: Local Payment Methods

1. Create `ideal.ts` — iDEAL: issuer list, redirect flow, return callback
2. Create `sepa.ts` — SEPA: IBAN collection, mandate, async settlement (2-3 days)
3. Create `boleto.ts` — Boleto: voucher generation, PDF, delayed confirmation (3 days)

### Step 4: Payment Router

1. Create `paymentRouter.ts`:
   - `selectProvider(region, paymentMethod): PaymentProvider`
   - EU customers with iDEAL/SEPA → Adyen
   - Brazil customers → Adyen (Boleto)
   - All others → Stripe (default)
   - Override: merchant can force provider via config

2. Integrate router into checkout flow:
   - Replace direct Stripe calls with router-selected provider
   - Pass provider selection through to confirmation page

### Step 5: Webhook Handling

1. Add Adyen webhook endpoint to `payments.ts`:
   - `POST /webhooks/adyen` — receives payment notifications
   - HMAC signature verification
   - Map notification types to order status updates
   - Idempotency via notification reference

### Step 6: Monitoring & Alerts

1. Add payment metrics by provider: success/failure rates, processing time
2. Alert on provider-specific failure spikes
3. Dashboard: side-by-side Stripe vs Adyen performance
