# Plan: Test Data Generator for Integration Test Suite

## Context

Our integration test suite relies on hand-crafted JSON fixtures in `test/fixtures/`. These are brittle, hard to maintain, and don't cover edge cases well. We need a test data generator that produces realistic fixture data conforming to our domain schemas, so the downstream test runner (`test/runner.ts`) can consume them for integration tests.

## Approach

Build a schema-driven test data generator that reads our TypeScript interfaces, generates randomized but valid fixture data using `faker.js`, and writes fixture files that the existing test runner picks up. The generator supports deterministic seeding for reproducible test runs.

## Files to Modify

- `src/test-gen/generator.ts` тАФ core fixture generation engine (new)
- `src/test-gen/schema-parser.ts` тАФ TypeScript interface parser (new)
- `src/test-gen/profiles.ts` тАФ predefined generation profiles (new)
- `src/test-gen/writers/json-writer.ts` тАФ JSON fixture output (new)
- `src/test-gen/writers/csv-writer.ts` тАФ CSV fixture output (new)
- `test/fixtures/.gitkeep` тАФ generated fixtures directory
- `scripts/generate-fixtures.sh` тАФ CLI wrapper script (new)
- `package.json` тАФ add `@faker-js/faker`, `ts-morph`

## Implementation

### Phase 1: Schema Parsing

1. Install dependencies:
   ```
   npm install @faker-js/faker ts-morph
   ```

2. Create `src/test-gen/schema-parser.ts`:
   - Use `ts-morph` to parse TypeScript interface files from `src/types/`
   - Extract field names, types, and JSDoc annotations
   - Support nested interfaces and array types
   - Recognize constraint annotations: `@min`, `@max`, `@pattern`, `@enum`
   - Build an intermediate schema representation (ISR) for each interface

3. Define the ISR types in `src/test-gen/types.ts`:
   ```typescript
   interface FieldSchema {
     name: string;
     type: 'string' | 'number' | 'boolean' | 'date' | 'array' | 'object';
     constraints?: {
       min?: number;
       max?: number;
       pattern?: string;
       enum?: string[];
       nullable?: boolean;
     };
     children?: FieldSchema[];  // for nested objects and arrays
   }
   ```

### Phase 2: Data Generation Engine

4. Create `src/test-gen/generator.ts`:
   - Accept an ISR and generation options (count, seed, profile)
   - Use `faker.js` seeded with deterministic seed for reproducibility
   - Generate field values matching the ISR constraints
   - Support cross-field dependencies (e.g., `endDate > startDate`)
   - Handle circular references in nested schemas (max depth = 3)

5. Implement generation strategies per field type:
   - `string` тАФ faker method selection based on field name heuristics
     (e.g., `email` field uses `faker.internet.email()`)
   - `number` тАФ respect `min`/`max`, use appropriate distribution
   - `boolean` тАФ configurable true/false ratio per profile
   - `date` тАФ generate within a configurable time range
   - `array` тАФ generate 1-5 items by default, configurable
   - `object` тАФ recurse into child schema

6. Create `src/test-gen/profiles.ts` with predefined profiles:
   - `minimal` тАФ required fields only, smallest valid values
   - `typical` тАФ realistic data with common patterns
   - `edge` тАФ boundary values, max lengths, special characters
   - `stress` тАФ large arrays, deeply nested objects, Unicode

### Phase 3: Fixture Output

7. Create `src/test-gen/writers/json-writer.ts`:
   - Write generated data as JSON fixture files to `test/fixtures/`
   - One file per schema: `test/fixtures/{schema-name}.json`
   - Include metadata header with generation timestamp and seed
   - Pretty-print with 2-space indentation

8. Create `src/test-gen/writers/csv-writer.ts`:
   - Flatten nested objects with dot-notation column names
   - Write header row followed by data rows
   - Handle arrays by creating separate relational CSV files
   - Proper escaping of special characters

9. Create `scripts/generate-fixtures.sh`:
   ```bash
   #!/bin/bash
   set -eo pipefail
   SEED="${1:-42}"
   PROFILE="${2:-typical}"
   COUNT="${3:-50}"
   npx ts-node src/test-gen/generator.ts \
     --seed "$SEED" \
     --profile "$PROFILE" \
     --count "$COUNT" \
     --output test/fixtures/
   echo "Generated $COUNT fixtures per schema (seed=$SEED, profile=$PROFILE)"
   ```

### Phase 4: Schema Coverage

10. Add generators for all domain schemas:
    - `User` тАФ id, email, name, role, createdAt, preferences (nested)
    - `Order` тАФ id, userId, items (array), total, status, timestamps
    - `Product` тАФ id, sku, name, price, inventory, categories (array)
    - `Payment` тАФ id, orderId, amount, currency, method, status
    - `AuditLog` тАФ id, userId, action, resource, metadata (object), timestamp

11. Implement relationship generation:
    - Orders reference valid User IDs from generated User fixtures
    - Payments reference valid Order IDs from generated Order fixtures
    - Ensure referential integrity across all generated fixture files

12. Add a `manifest.json` output that lists all generated fixture files:
    ```json
    {
      "generated_at": "2024-01-15T10:30:00Z",
      "seed": 42,
      "profile": "typical",
      "fixtures": {
        "users": { "file": "users.json", "count": 50 },
        "orders": { "file": "orders.json", "count": 50 },
        "products": { "file": "products.json", "count": 50 }
      }
    }
    ```

### Phase 5: CI Integration

13. Add `generate-fixtures` npm script to `package.json`:
    ```json
    {
      "scripts": {
        "generate-fixtures": "bash scripts/generate-fixtures.sh",
        "test:integration": "npm run generate-fixtures && jest --config jest.integration.js"
      }
    }
    ```

14. Update `.github/workflows/ci.yml` to generate fixtures before integration tests:
    - Cache generated fixtures by seed value
    - Run with `edge` profile in addition to `typical` for broader coverage

15. Add a `--validate` flag to the generator that checks output against
    the original TypeScript interfaces using runtime validation.

- Run with `edge` profile тАФ boundary values present (empty strings, max ints)
- Generated User fixtures have valid email formats and enum roles
- Order fixtures reference existing User IDs from user fixtures
- `npm run test:integration` consumes generated fixtures and passes
- Manifest file accurately lists all generated fixtures with counts
