# Plan: Node.js API Auth Refactor — JWT to Session Tokens

## Context

Our Node.js API server currently uses JWT tokens for authentication. JWTs are causing
issues: they can't be revoked without a blocklist, the tokens are large (inflating
request headers), and we've had two incidents where leaked JWTs were used for hours
before expiry. We're migrating to opaque session tokens stored server-side with
Redis-backed session storage.

## Current State

- Express 4.x API with 32 authenticated endpoints
- JWT issued on login, verified via middleware on every request
- Token expiry set to 24 hours, no refresh token flow
- User roles embedded in JWT payload: `admin`, `editor`, `viewer`

## Approach

Replace JWT issuance and verification with opaque session tokens. On login, generate
a cryptographically random token, store the session data in Redis, and return the token
to the client. On each request, look up the token in Redis to retrieve the session.
This gives us instant revocation, smaller tokens, and server-side control over session
data.

## Files to Modify

- Update the authentication handler to issue session tokens instead of JWTs
- Modify the config parser to add Redis connection settings
- Update the auth middleware to validate session tokens against Redis
- Modify the login endpoint to create sessions
- Update the logout endpoint to destroy sessions
- Add a session cleanup job for expired entries
- Modify the role-checking middleware to read roles from session data
- Update the user profile endpoint that currently decodes the JWT

## Implementation

### Phase 1: Session Infrastructure

1. Install `ioredis` and `crypto` dependencies
2. Create a session manager module that handles token generation and storage
3. Configure Redis connection using environment variables
4. Implement `createSession(userId, roles, ttl)` — generates 32-byte hex token,
   stores `{userId, roles, createdAt, expiresAt}` in Redis with TTL
5. Implement `getSession(token)` — retrieves session data from Redis
6. Implement `destroySession(token)` — deletes the Redis key
7. Add connection health check with retry logic

### Phase 2: Auth Flow Migration

1. Update the login handler to call `createSession()` instead of `jwt.sign()`
2. Return the opaque token in the response body (same field name for backward compat)
3. Update the authentication middleware to call `getSession()` instead of `jwt.verify()`
4. If session not found or expired, return 401
5. Attach session data to `req.session` instead of `req.user`
6. Update the role-checking middleware to read `req.session.roles` instead of
   decoding roles from the JWT
7. Update the logout handler to call `destroySession()`

### Phase 3: Endpoint Updates

1. Update all endpoints that reference `req.user` to use `req.session`
2. Update the profile endpoint — it currently calls `jwt.decode()` to show token
   metadata; replace with session data lookup
3. Add a `GET /api/sessions/active` admin endpoint listing active sessions
4. Add a `DELETE /api/sessions/:token` admin endpoint for forced revocation
5. Update API documentation for the new token format

### Phase 4: Cleanup & Migration Safety

1. Add a background job that runs every hour to clean up expired sessions in Redis
2. Implement a dual-mode transition period: accept both JWT and session tokens for
   two weeks, with a feature flag to control which mode is active
3. Add monitoring: log session creation/destruction rates
4. Remove JWT dependencies (`jsonwebtoken` package) after transition period
5. Update environment variable documentation

## Verification

1. Unit tests for session manager: create, get, destroy, expiry
2. Integration tests for login → authenticated request → logout flow
3. Verify 401 response when using a destroyed session token
4. Test role-based access control works with session-stored roles
5. Load test Redis session lookups under concurrent requests
6. Verify dual-mode accepts both token types during transition
7. Confirm leaked token can be revoked immediately via admin endpoint
