# Plan: User Notification System ‚Äî Preferences, API, and UI

## Context

Users currently receive all notification types (order updates, promotions, system alerts)
with no way to control preferences. We're getting complaints about notification spam.
We need a notification preferences system that lets users choose which channels (email,
SMS, push) they want for each notification category, plus a notification center in the
frontend showing recent notifications.

## Current State

- Notifications sent via a monolithic `NotificationSender` class
- No user preference storage ‚Äî everyone gets everything
- Email via SendGrid, SMS via Twilio, push via Firebase
- ~15 notification types across 4 categories: orders, promotions, account, system

## Approach

Build in three phases: (1) database schema for preferences with sensible defaults,
(2) API endpoints for reading and updating preferences plus a notification dispatch
service that respects them, (3) frontend notification center and preference management
UI. Each phase is committed independently so we can ship incrementally.

## Implementation

### Phase 1: Schema & Preference Service

Phase 1 establishes the data model and commits the schema migration so the database
is ready for API development. We commit at the end of this phase.

1. Create `004_notification_preferences.sql`:
   - `notification_categories` table: `id`, `name`, `description`, `default_channels`
   - Seed 4 categories: orders, promotions, account, system
   - `user_notification_preferences` table: `user_id`, `category_id`, `email_enabled`,
     `sms_enabled`, `push_enabled`
   - Default: all channels enabled for orders/account/system, email-only for promotions
   - Unique constraint on `(user_id, category_id)`
2. Create `005_notification_log.sql`:
   - `notification_log` table: `id`, `user_id`, `category_id`, `channel`, `subject`,
     `body_preview`, `status`, `sent_at`, `read_at`
   - Index on `(user_id, sent_at DESC)` for inbox queries
   - Index on `(user_id, read_at)` for unread count
3. Create `types.ts` with `NotificationCategory`, `Channel`, `UserPreference` types
4. Create `PreferenceService.ts`:
   - `getPreferences(userId)` ‚Äî returns all category preferences, using defaults for
     any missing rows
   - `updatePreference(userId, categoryId, channels)` ‚Äî upsert preference
   - `getEffectiveChannels(userId, categoryId)` ‚Äî returns enabled channels for dispatch
5. Run migrations, verify schema, commit Phase 1

### Phase 2: API & Dispatch Service

Phase 2 builds the API layer and the preference-aware dispatcher. This phase is
committed after all endpoints are working.

1. Create `Dispatcher.ts`:
   - `dispatch(userId, categoryId, notification)` ‚Äî looks up effective channels,
     sends via each enabled channel
   - Delegates to existing SendGrid/Twilio/Firebase integrations
   - Logs every sent notification to `notification_log`
   - Handles channel failures independently ‚Äî if email fails, still try SMS/push
2. Refactor `NotificationSender.ts` to call `Dispatcher.dispatch()` instead of
   sending directly
3. Create `preferences.ts` API routes:
   - `GET /api/users/:id/preferences` ‚Äî list all category preferences
   - `PUT /api/users/:id/preferences/:categoryId` ‚Äî update channels for a category
   - `POST /api/users/:id/preferences/reset` ‚Äî reset to defaults
4. Create `notifications.ts` API routes:
   - `GET /api/users/:id/notifications` ‚Äî paginated notification history
   - `GET /api/users/:id/notifications/unread-count` ‚Äî badge count
   - `PATCH /api/users/:id/notifications/:notifId/read` ‚Äî mark as read
   - `POST /api/users/:id/notifications/mark-all-read` ‚Äî bulk mark read
5. Add request validation and auth middleware to all endpoints
6. Commit Phase 2

### Phase 3: Frontend Components

Phase 3 builds the user-facing UI. Committed after component integration is verified.

1. Create `NotificationCenter.tsx`:
   - Bell icon in header with unread count badge
   - Dropdown panel showing recent 20 notifications
   - Click notification to mark as read and navigate to relevant page
   - "Mark all as read" button
   - "View all" link to full notification history page
2. Create `PreferencePanel.tsx`:
   - Grid layout: rows = categories, columns = channels
   - Toggle switches for each cell
   - "System" category toggles are disabled (always enabled, shown as info)
   - Save button calls `PUT /api/users/:id/preferences/:categoryId`
   - Show toast on save success/failure
3. Integrate `NotificationCenter` into the app header layout
4. Add `PreferencePanel` to the user settings page
5. Wire up real-time unread count via polling every 30 seconds
6. Add loading skeletons for async data fetching
7. Add empty states: "No notifications yet" with illustration
8. Commit Phase 3
