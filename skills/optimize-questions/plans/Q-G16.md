# Plan: Order Processing State Machine

## Context

Our e-commerce platform processes orders through a multi-step workflow: from placement
through payment, fulfillment, shipping, and delivery. Currently this logic is spread
across 6 different service files with ad-hoc status updates. Race conditions have caused
orders to skip states (e.g., jumping from "placed" to "shipped" without payment
confirmation). We need a formal state machine to enforce valid transitions.

## Current State

- Order status stored as a string column in the `orders` table
- Status updates happen via direct SQL in various service methods
- Known states: `placed`, `payment_pending`, `paid`, `fulfilling`, `shipped`, `delivered`
- Also: `cancelled`, `refunded` (can occur from multiple states)
- No transition validation — any service can set any status

## Approach

Build a `StateMachine` class that encodes all valid transitions and enforces them at
the application layer. Each state transition triggers registered side effects (send
email, update inventory, notify warehouse). The state machine is the single source of
truth for order status changes — no direct SQL updates to the status column.

## Files to Modify

- `src/orders/StateMachine.ts` (new) — core state machine engine
- `src/orders/transitions.ts` (new) — transition definitions and side effects
- `src/orders/OrderService.ts` — refactor to use state machine
- `src/orders/PaymentService.ts` — refactor to use state machine
- `src/orders/FulfillmentService.ts` — refactor to use state machine
- `src/orders/ShippingService.ts` — refactor to use state machine
- `src/orders/types.ts` — add state and transition types
- `test/state-machine.test.ts` (new) — transition validation tests

## Implementation

### Phase 1: State Machine Core

1. Define the `OrderState` enum in `types.ts`:
   `PLACED | PAYMENT_PENDING | PAID | FULFILLING | SHIPPED | DELIVERED | CANCELLED | REFUNDED`
2. Define the `TransitionEvent` enum:
   `INITIATE_PAYMENT | CONFIRM_PAYMENT | PAYMENT_FAILED | BEGIN_FULFILLMENT | SHIP | DELIVER | CANCEL | REFUND`
3. Create `StateMachine` class with:
   - `currentState: OrderState`
   - `transition(event: TransitionEvent): OrderState` — validates and applies
   - `canTransition(event: TransitionEvent): boolean`
   - `getAvailableEvents(): TransitionEvent[]`
4. Store transition table as a `Map<OrderState, Map<TransitionEvent, OrderState>>`
5. Throw `InvalidTransitionError` for illegal transitions

### Phase 2: Transition Definitions

1. Create `transitions.ts` with the full transition map:
   - `PLACED` + `INITIATE_PAYMENT` -> `PAYMENT_PENDING`
   - `PAYMENT_PENDING` + `CONFIRM_PAYMENT` -> `PAID`
   - `PAYMENT_PENDING` + `PAYMENT_FAILED` -> `PLACED`
   - `PAID` + `BEGIN_FULFILLMENT` -> `FULFILLING`
   - `FULFILLING` + `SHIP` -> `SHIPPED`
   - `SHIPPED` + `DELIVER` -> `DELIVERED`
   - `PLACED` + `CANCEL` -> `CANCELLED`
   - `PAYMENT_PENDING` + `CANCEL` -> `CANCELLED`
   - `PAID` + `CANCEL` -> `CANCELLED`
   - `PAID` + `REFUND` -> `REFUNDED`
   - `DELIVERED` + `REFUND` -> `REFUNDED`
2. Register side effects per transition:
   - `CONFIRM_PAYMENT`: update payment record, send receipt email
   - `BEGIN_FULFILLMENT`: reserve inventory, notify warehouse
   - `SHIP`: record tracking number, send shipping notification
   - `DELIVER`: send delivery confirmation, trigger review request
   - `CANCEL`: release inventory, process refund if paid
   - `REFUND`: initiate payment reversal, send refund confirmation
3. Side effects execute after successful state transition

### Phase 3: Service Refactor

1. Refactor `OrderService.ts`:
   - Load order, instantiate `StateMachine` with current state
   - Call `transition(event)` instead of direct status SQL update
   - Persist new state to database after successful transition
2. Refactor `PaymentService.ts`:
   - On payment webhook: load order machine, fire `CONFIRM_PAYMENT` or `PAYMENT_FAILED`
3. Refactor `FulfillmentService.ts`:
   - On fulfillment start: fire `BEGIN_FULFILLMENT`
   - On packing complete: fire `SHIP` with tracking data
4. Refactor `ShippingService.ts`:
   - On delivery confirmation: fire `DELIVER`
5. Add database transaction wrapping: state change + side effects in one transaction
6. Add audit log: record every transition with timestamp, actor, old/new state

## Verification

1. Unit tests for every valid transition in the map
2. Unit tests for every invalid transition (expect `InvalidTransitionError`)
3. Test side effect execution: mock email/inventory services, verify calls
4. Integration test: full order lifecycle from PLACED to DELIVERED
5. Integration test: cancel from each cancellable state
6. Test concurrent transitions: two threads try to transition same order
7. Verify audit log records all transitions with correct metadata
