# Plan: Simplify User API Response Payload

## Context

Our public REST API returns bloated user objects with 25+ fields, many of which are
internal-only (audit timestamps, internal IDs, permission bitmasks). Front-end teams
have requested leaner payloads. Security audit also flagged that internal fields like
`passwordHash` are being filtered client-side rather than server-side.

## Current State

- Express 4.18, TypeScript 5.3
- `GET /api/v2/users/:id` returns full user object
- `GET /api/v2/users` returns array of full user objects
- Public API consumers: mobile app (iOS/Android), partner integrations (3 partners),
  internal dashboard
- Current response shape includes: `id`, `name`, `email`, `phone`, `avatar`,
  `role`, `permissions`, `createdAt`, `updatedAt`, `lastLoginAt`, `passwordHash`,
  `internalNotes`, `auditTrail`, `permissionBitmask`
- Partners have built integrations relying on the current response shape
- API versioned at URL level: `/api/v2/...`

## Approach

Remove unnecessary fields from user API responses to reduce payload size and eliminate
security concerns. Create a serialization layer that explicitly whitelists fields for
the API response.

## Files to Modify

- `src/serializers/userSerializer.ts` (new) — whitelist-based field serializer
- `src/routes/v2/users.ts` — apply serializer to all user endpoints
- `src/types/api/userResponse.ts` (new) — public user response type
- `src/middleware/fieldSelector.ts` (new) — sparse fieldset support (?fields=name,email)

## Implementation

### Phase 1: Serializer

1. Define public user response type in `src/types/api/userResponse.ts`:
   ```typescript
   interface PublicUser {
     id: string;
     name: string;
     phone: string | null;
     avatar: string | null;
     role: string;
     createdAt: string;
   }
   ```

2. Create `userSerializer.ts`:
   - `serializeUser(user: User): PublicUser` — maps internal User to public shape
   - `serializeUsers(users: User[]): PublicUser[]` — batch serialization
   - Explicitly whitelist only the fields in `PublicUser`
   - Format dates to ISO-8601 strings
   - Null-safe: handle missing optional fields

3. Remove these fields from responses:
   - `email` — removed, available only via dedicated `/users/:id/email` endpoint
   - `passwordHash` — security risk, never should have been included
   - `internalNotes` — internal only
   - `auditTrail` — internal only
   - `permissionBitmask` — internal only
   - `permissions` — replaced by role string
   - `updatedAt` — internal tracking only
   - `lastLoginAt` — internal tracking only

### Phase 2: Sparse Fieldsets

1. Create `fieldSelector.ts` middleware:
   - Parse `?fields=name,avatar,role` query parameter
   - Filter serialized response to only requested fields
   - Always include `id` regardless of field selection
   - Validate requested fields against `PublicUser` type

2. Apply middleware to `GET /api/v2/users` and `GET /api/v2/users/:id`

### Phase 3: Integration

1. Update `src/routes/v2/users.ts`:
   - Import and apply `serializeUser` to single-user endpoints
   - Import and apply `serializeUsers` to list endpoints
   - Apply `fieldSelector` middleware to both routes

2. Add response compression for large user lists

3. Update OpenAPI spec to reflect new response shape

## Verification

1. Unit test serializer: verify only whitelisted fields appear in output
2. Unit test: `passwordHash` and `internalNotes` never present in serialized output
3. Unit test sparse fieldsets: `?fields=name` returns only `id` and `name`
4. Integration test: fetch user via API, verify response matches `PublicUser` shape
5. Verify OpenAPI spec matches actual response
6. Load test: compare response sizes before and after (target 60% reduction)
