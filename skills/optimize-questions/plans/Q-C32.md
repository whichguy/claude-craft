# Plan: Add User Search Feature

## Context

Our admin dashboard needs a user search feature. Admins currently browse users via
paginated tables but cannot search by name, email, or role. The user base has grown
to 2.3 million records and admins report spending significant time scrolling through
pages to find specific users.

## Current State

- Node.js 20, TypeScript 5.3, PostgreSQL 15
- `src/repositories/userRepository.ts` has `getAll(): Promise<User[]>` and `getById(id)`
- User table: 2.3M rows, ~1.2KB per row average
- No full-text search indexes
- Admin dashboard: React 18 with server-side data fetching
- Current pagination: client-side (fetch all, paginate in browser) — already slow

## Approach

Add a search endpoint that queries users by name, email, or role. The search will
use the existing `getAll()` method to load users and filter in application code for
flexibility. This avoids complex SQL queries and keeps the filtering logic testable
in isolation.

## Files to Modify

- `src/services/userSearchService.ts` (new) — search logic
- `src/routes/admin/users.ts` — add search endpoint
- `src/repositories/userRepository.ts` — no changes needed (use existing getAll)
- `src/types/search.ts` (new) — search request/response types
- `src/utils/textMatch.ts` (new) — fuzzy text matching utility

## Implementation

### Phase 1: Search Types & Utilities

1. Define search types in `src/types/search.ts`:
   ```typescript
   interface SearchQuery {
     term: string;
     fields: ('name' | 'email' | 'role')[];
     caseSensitive?: boolean;
   }
   interface SearchResult {
     users: User[];
     total: number;
     took: number; // milliseconds
   }
   ```

2. Create `textMatch.ts` with fuzzy matching:
   - `fuzzyMatch(haystack, needle)` — substring + Levenshtein distance
   - `scoreMatch(user, query)` — composite relevance score across fields
   - Configurable similarity threshold (default 0.7)

### Phase 2: Search Service

1. Create `userSearchService.ts`:
   - `search(query: SearchQuery): Promise<SearchResult>`
   - Load all users via `userRepository.getAll()`
   - Apply `textMatch` filter against requested fields
   - Sort results by relevance score (highest first)
   - Return matching users with total count and timing

2. Add result caching:
   - Cache `getAll()` result for 60 seconds to avoid repeated DB loads
   - Invalidate cache on user create/update/delete
   - Cache key: `users:all`

3. Add search logging:
   - Log search term, result count, and time taken
   - Track popular search terms for analytics

### Phase 3: API Endpoint

1. Add `GET /api/admin/users/search?q=term&fields=name,email` to `users.ts`
2. Parse and validate query parameters
3. Rate limit: 30 searches per minute per admin
4. Return search results with total count and timing metadata

### Phase 4: Dashboard Integration

1. Add search input component to admin user list page
2. Debounce search input (300ms) to avoid excessive API calls
3. Highlight matching text in search results
4. Show "X results found in Yms" summary

## Verification

1. Unit test `textMatch`: exact match, partial match, fuzzy match, case sensitivity
2. Unit test search service: filter by name, email, role combinations
3. Unit test result sorting by relevance score
4. Integration test: create test users, search by name, verify correct results
5. Test cache invalidation: update a user, verify next search reflects changes
6. Performance test: search with 1000 test users, verify response under 500ms
