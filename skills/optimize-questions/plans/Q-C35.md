# Plan: Automated Plan Review System

## Context

Our team reviews implementation plans before coding begins. Currently this is manual —
a senior engineer reads the plan and writes feedback. We want to automate this with
an LLM-based evaluator agent that checks plans against our quality criteria. The review
covers architecture, testing strategy, security, performance, maintainability, and more.

## Current State

- TypeScript 5.3, Node.js 20
- Plans are markdown files, typically 60-120 lines
- Review criteria: 38 questions organized into 5 categories (impact, testing, security,
  performance, maintainability)
- Current manual review takes 30-45 minutes per plan
- Average plan complexity: moderate to high (3-4 phases, 8-12 files)
- Some plans exceed 800 lines for large features

## Approach

Build an evaluator agent that takes a plan and the full question set, then produces a
structured review with pass/fail per question, evidence citations, and an overall score.
A single evaluator pass will assess all questions to keep the system simple.

## Files to Modify

- `src/agents/planEvaluator.ts` (new) — main evaluator agent
- `src/agents/prompts/evaluatorPrompt.ts` (new) — system prompt for the evaluator
- `src/types/review.ts` (new) — review result types
- `src/questions/questionSet.ts` (new) — load and format the 38 questions
- `src/review/reviewRunner.ts` (new) — orchestration: load plan, run evaluator, format output
- `src/review/reportFormatter.ts` (new) — convert raw results to markdown report

## Implementation

### Phase 1: Question & Type Setup

1. Define review types in `src/types/review.ts`:
   ```typescript
   interface QuestionResult {
     questionId: string;
     category: string;
     question: string;
     verdict: 'pass' | 'fail' | 'partial' | 'not-applicable';
     evidence: string;
     suggestion?: string;
   }
   interface ReviewResult {
     planFile: string;
     timestamp: string;
     results: QuestionResult[];
     score: number;
     summary: string;
   }
   ```

2. Create `questionSet.ts`:
   - Load all 38 questions from markdown file
   - Format into structured prompt sections
   - Group by category for the evaluator prompt

### Phase 2: Evaluator Agent

1. Create `evaluatorPrompt.ts` with system prompt:
   - Role: expert software plan reviewer
   - Instructions: evaluate the plan against ALL provided questions
   - Output format: JSON array of QuestionResult objects
   - Emphasis: cite specific plan sections as evidence

2. Create `planEvaluator.ts`:
   - `evaluate(planContent: string, questions: Question[]): Promise<QuestionResult[]>`
   - Construct prompt: system prompt + full plan content + all 38 questions
   - Single LLM call with the complete context
   - Parse structured JSON response
   - Validate all 38 questions have corresponding results

3. Implement the review as a single pass:
   - Concatenate all questions into one prompt section
   - Include full plan text (up to 800+ lines)
   - Ask evaluator to assess all 38 questions in one response
   - Parse the combined output into individual QuestionResult entries

### Phase 3: Orchestration & Reporting

1. Create `reviewRunner.ts`:
   - `reviewPlan(planPath: string): Promise<ReviewResult>`
   - Load plan from file system
   - Load question set
   - Run evaluator
   - Calculate overall score (pass=1, partial=0.5, fail=0, N/A excluded)
   - Return structured result

2. Create `reportFormatter.ts`:
   - Convert ReviewResult to markdown report
   - Group results by category
   - Show pass/fail badges per question
   - Summary section with score and top issues
   - Evidence citations with plan line references

3. Add CLI entry point:
   - `npx review-plan <plan-file>` runs the full pipeline
   - Output to stdout or `--output report.md`

## Verification

1. Unit test question loading: verify all 38 questions parsed correctly
2. Unit test report formatter: verify markdown structure and grouping
3. Unit test score calculation: verify pass/partial/fail/N/A weighting
4. Integration test: run evaluator on a known-good plan, verify high score
5. Integration test: run on known-bad plan, verify failures detected
6. Measure token usage: verify single-pass stays within model context limits
