# Plan: Database Connection Pooling Optimization

## Context

Our Node.js API service (`src/server.ts`) currently opens a new PostgreSQL connection per request, causing latency spikes under load. We need to implement connection pooling with configurable pool sizes per environment. The database configuration lives in `config/database.yml` and our ORM models are in `src/models/`.

## Approach

Introduce `pg-pool` as the connection layer, configure pool sizing via `config/database.yml`, and update the User model to use pooled connections. We will add health-check endpoints that report pool utilization metrics.

## Files to Modify

- `config/database.yml` — add pool configuration block
- `src/models/User.ts` — switch from single-connection to pooled queries
- `src/db/pool.ts` — new file, pool initialization and export
- `src/routes/health.ts` — add pool stats endpoint
- `package.json` — add `pg-pool` dependency

## Implementation

### Phase 1: Configuration and Pool Setup

1. Install `pg-pool` dependency:
   ```
   npm install pg-pool @types/pg-pool
   ```

2. Update `config/database.yml` to add pool settings under each environment:
   ```yaml
   production:
     host: ${DB_HOST}
     port: 5432
     pool:
       min: 5
       max: 20
       idle_timeout_ms: 30000
       connection_timeout_ms: 5000
   ```

3. Create `src/db/pool.ts`:
   - Read `config/database.yml` using `js-yaml`
   - Instantiate `Pool` with the parsed config values
   - Export the pool singleton
   - Add graceful shutdown hook (`process.on('SIGTERM')`)

### Phase 2: Model Migration

4. Update `src/models/User.ts` to import the pool from `src/db/pool.ts` instead of
   creating its own `Client` instance.

5. Replace all `client.query()` calls with `pool.query()` calls in User.ts. The
   `findById`, `findByEmail`, and `updateProfile` methods all need updating.

6. Change the `User.transaction()` helper to check out a client from the pool,
   run the transaction, and release back to pool in a `finally` block.

7. Update the `User.bulkInsert()` method to use `pool.connect()` for dedicated
   client access during COPY operations, ensuring the client is released after.

### Phase 3: Health and Monitoring

8. Add a `/health/db` route in `src/routes/health.ts` that returns:
   ```json
   {
     "total": 20,
     "idle": 15,
     "waiting": 0,
     "active": 5
   }
   ```

9. Add a Prometheus-compatible `/metrics` endpoint that exposes:
   - `db_pool_size` gauge
   - `db_pool_active` gauge
   - `db_pool_waiting` gauge
   - `db_pool_idle` gauge

10. Wire the health route into the Express app in `src/server.ts`.

### Phase 4: Cleanup

11. Remove the legacy `src/db/connection.ts` single-connection module.

12. Update `src/models/index.ts` barrel export to use the new pool module.

13. Update environment-specific configs in `config/database.yml` for staging
    and development environments with appropriate pool sizes.

## Verification

- Run `npm test` — all existing model tests should pass without modification
  since query interfaces remain the same.
- Load test with `autocannon -c 100 -d 30 http://localhost:3000/api/users` and
  confirm no connection errors.
- Hit `/health/db` and verify pool stats reflect expected idle/active counts.
- Confirm graceful shutdown releases all connections (check PostgreSQL
  `pg_stat_activity` after SIGTERM).
