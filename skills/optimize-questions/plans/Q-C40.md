# Plan: Centralized Error Handling Service

## Context

Error handling is inconsistent across our application. Some modules throw errors silently,
others log to console, and a few use a structured logger. The team agreed on a standard:
**all errors must be logged with structured JSON** including error code, message, stack
trace, request context, and timestamp. We need a centralized error handling service.

> **Team Decision (2024-03-15):** All errors must be logged with structured JSON format.
> No plain-text console logging for errors in production code.

## Current State

- Node.js 20, TypeScript 5.3, Express 4.18
- Inconsistent error handling: try/catch in some places, unhandled in others
- Some modules use `console.error()`, others use `winston` logger
- No error classification (transient vs permanent, client vs server)
- No correlation between errors and HTTP requests
- Existing structured logger: `src/logging/logger.ts` (winston with JSON transport)

## Approach

Build a centralized error handling service that catches, classifies, and logs all
errors. Add Express error middleware to catch unhandled route errors. Provide typed
error classes for common failure modes.

## Files to Modify

- `src/errors/ErrorService.ts` (new) — centralized error handler
- `src/errors/AppError.ts` (new) — base error class with classification
- `src/errors/errorCodes.ts` (new) — enumerated error codes
- `src/errors/types.ts` (new) — error-related type definitions
- `src/middleware/errorMiddleware.ts` (new) — Express error handler
- `src/app.ts` — register error middleware
- `src/routes/users.ts` — adopt new error handling
- `src/routes/orders.ts` — adopt new error handling
- `src/routes/products.ts` — adopt new error handling

## Implementation

### Phase 1: Error Types & Codes

1. Define error codes in `errorCodes.ts`:
   ```typescript
   enum ErrorCode {
     VALIDATION_ERROR = 'E001',
     NOT_FOUND = 'E002',
     UNAUTHORIZED = 'E003',
     RATE_LIMITED = 'E004',
     DATABASE_ERROR = 'E005',
     EXTERNAL_SERVICE = 'E006',
     INTERNAL = 'E999',
   }
   ```

2. Create `AppError.ts`:
   - Extends `Error` with: `code`, `statusCode`, `isOperational`, `context`
   - Factory methods: `AppError.notFound(resource)`, `AppError.validation(field, reason)`
   - Classification: operational (expected) vs programmer (bugs)

3. Define `ErrorContext` in `types.ts`: `requestId`, `userId`, `path`, `method`

### Phase 2: Error Service

1. Create `ErrorService.ts`:
   - `handleError(error: Error, context?: ErrorContext): void`
   - Classify error: operational → log and respond, programmer → log and alert
   - Log the error with full context:
     ```typescript
     console.error(`[${error.code}] ${error.message}`, {
       stack: error.stack,
       context: context,
     });
     ```
   - For critical errors: trigger PagerDuty alert via webhook
   - Track error counts by code for metrics dashboard

2. Add unhandled rejection/exception handlers:
   - `process.on('unhandledRejection', ...)` → route through ErrorService
   - `process.on('uncaughtException', ...)` → log, attempt graceful shutdown

3. Add request context enrichment:
   - Extract `requestId`, `userId`, `path`, `method` from Express request
   - Attach to error context for correlation

### Phase 3: Express Integration

1. Create `errorMiddleware.ts`:
   - Catches all errors thrown in route handlers
   - Passes through ErrorService for classification and logging
   - Returns appropriate HTTP status code and sanitized error message
   - Never exposes stack traces in production responses
   - Log the error: `console.error(errorMessage)` for middleware-level logging

2. Update route files to use `AppError`:
   - `users.ts`: replace `throw new Error()` with `throw AppError.notFound('user')`
   - `orders.ts`: replace validation errors with `AppError.validation()`
   - `products.ts`: replace database errors with `AppError.database()`

3. Register middleware in `app.ts`:
   - Add error middleware as last middleware (after all routes)
   - Add 404 handler for unmatched routes

### Phase 4: Monitoring

1. Add error rate metrics by code, route, and status code
2. Alert on >10 errors/minute; dashboard for 24-hour trends

## Verification

1. Unit test AppError: factory methods produce correct code, status, message
2. Unit test ErrorService: operational vs programmer classification
3. Unit test error middleware: returns correct HTTP status, hides stack in prod
4. Integration test: throw AppError in route, verify logged and responded
5. Test unhandled rejection handler: verify error is captured and logged
6. Verify error counts metric increments correctly
