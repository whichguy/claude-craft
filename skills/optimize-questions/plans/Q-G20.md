# Plan: Structured Logging Infrastructure

## Files to Modify

- `src/logger.ts` — new structured logger module
- `src/middleware/request-logger.ts` — HTTP request logging middleware
- `src/server.ts` — wire middleware and replace console.log calls
- `src/services/payment.ts` — add structured logging to payment flow
- `src/services/auth.ts` — add structured logging to auth flow
- `package.json` — add `pino` and `pino-pretty`

## Implementation

### Phase 1: Logger Module

1. Install dependencies:
   ```
   npm install pino pino-pretty pino-http
   ```

2. Create `src/logger.ts`:
   - Configure pino with JSON output in production, pretty-print in development
   - Set log level from `LOG_LEVEL` env var, default `info`
   - Add `service`, `version`, and `environment` as base fields
   - Export a `createChildLogger(context)` factory for per-module loggers

3. Set up log level hierarchy: `trace < debug < info < warn < error < fatal`.

### Phase 2: Request Logging Middleware

4. Create `src/middleware/request-logger.ts`:
   - Use `pino-http` to log every request with method, URL, status, and duration
   - Add request ID generation (UUID v4) and attach to `req.id`
   - Redact `authorization` header and `cookie` values from logs
   - Log response time in milliseconds

5. Register the middleware in `src/server.ts` before all route handlers.

### Phase 3: Service Integration

6. Update `src/services/payment.ts`:
   - Replace all `console.log` with structured logger calls
   - Add `paymentId`, `amount`, `currency` as structured fields
   - Log at `info` for successful charges, `error` for failures
   - Include `stripe_request_id` in payment log context

7. Update `src/services/auth.ts`:
   - Replace `console.log` and `console.error` with structured logger
   - Add `userId`, `action` (login/logout/refresh) as structured fields
   - Log failed login attempts at `warn` level with IP address
   - Log token refresh at `debug` level

8. Update `src/server.ts`:
   - Replace the 4 remaining `console.log` calls with the logger
   - Add startup log with port, environment, and node version
   - Add shutdown log in the SIGTERM handler

### Phase 4: Log Correlation

9. Implement correlation ID propagation:
   - Extract `X-Request-Id` header if present, otherwise generate new UUID
   - Store in AsyncLocalStorage for automatic propagation to child loggers
   - All service-level logs automatically include the correlation ID

10. Add `src/middleware/correlation.ts`:
    - Initialize AsyncLocalStorage store per request
    - Middleware runs before request-logger so ID is available everywhere

### Phase 5: Configuration

11. Add log rotation config for file transport in production:
    - Daily rotation, 14-day retention
    - Separate error log stream to `logs/error.log`
    - Combined log to `logs/combined.log`

12. Add environment variable documentation to `.env.example`:
    - `LOG_LEVEL` — minimum log level
    - `LOG_FORMAT` — json or pretty
    - `LOG_FILE_ENABLED` — enable file transport

## Verification

- Check that everything works.
- Confirm logs look right in production and development modes.
- Make sure the old console.log calls are all removed.
