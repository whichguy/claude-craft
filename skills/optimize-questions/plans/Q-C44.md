# Plan: Increase Payment Retry Limits

## Context

Payment processing currently retries failed charges 3 times with fixed 5-second delays.
Merchants report that transient bank errors often resolve within 30 seconds but our
retries are exhausted by then. Increasing to 5 retries with exponential backoff should
recover ~40% more failed payments.

## Current State

- `src/services/payment-processor.ts` — retry logic with MAX_RETRIES=3, DELAY=5000ms
- Processes ~2,000 payments/day, ~8% failure rate, ~60% of failures are transient

## Approach

Change retry parameters: MAX_RETRIES=5, use exponential backoff (2s, 4s, 8s, 16s, 32s)
instead of fixed delay. Add jitter to prevent thundering herd on recovery.

## Files to Modify

- `src/services/payment-processor.ts` — retry logic changes
- `src/config/payment.ts` — externalize retry config

## Implementation

1. Update `payment.ts` config:
   ```ts
   export const PAYMENT_RETRY = {
     maxRetries: 5,
     baseDelay: 2000,
     maxDelay: 32000,
     jitterRange: 500,
   };
   ```

2. Update retry loop in `payment-processor.ts`:
   ```ts
   for (let attempt = 0; attempt < config.maxRetries; attempt++) {
     try {
       return await chargeCard(paymentIntent);
     } catch (err) {
       if (!isTransient(err) || attempt === config.maxRetries - 1) throw err;
       const delay = Math.min(
         config.baseDelay * Math.pow(2, attempt) + Math.random() * config.jitterRange,
         config.maxDelay
       );
       await sleep(delay);
     }
   }
   ```

## Verification

1. Unit tests: verify 5 retries with exponential backoff timing
2. Unit tests: verify non-transient errors fail immediately
3. Integration test: mock transient failure on attempts 1-3, success on attempt 4
