# Plan: Add Notification System for Order Events

## Context

Our e-commerce platform needs to notify users about order status changes: confirmation,
shipping updates, delivery, and refunds. Currently there is no notification system.
Users find out about order updates only by manually checking the order status page.

## Current State

- Node.js 20, TypeScript 5.3
- Event-driven architecture with `src/events/eventBus.ts`:
  - `EventBus` class with `emit(event, payload)`, `on(event, handler)`, `off(event, handler)`
  - Events: `order.created`, `order.shipped`, `order.delivered`, `order.refunded`
  - Currently used by: inventory service (stock updates), analytics (tracking), audit log
- Order service emits events on every status transition
- No email/SMS/push infrastructure

## Approach

We will create a standalone notification module that independently tracks order events
and dispatches notifications. The module will maintain its own event listeners, its own
queue, and its own delivery logic.

## Files to Modify

- `src/notifications/notificationService.ts` (new) — core notification engine
- `src/notifications/notificationQueue.ts` (new) — in-memory notification queue
- `src/notifications/channels/emailChannel.ts` (new) — email delivery via SendGrid
- `src/notifications/channels/smsChannel.ts` (new) — SMS delivery via Twilio
- `src/notifications/channels/pushChannel.ts` (new) — push notification via Firebase
- `src/notifications/templateEngine.ts` (new) — notification message templates
- `src/notifications/eventListener.ts` (new) — listens to order events
- `src/types/notification.ts` (new) — notification types
- `src/app.ts` — initialize notification module on startup

## Implementation

### Phase 1: Notification Core

1. Define notification types in `src/types/notification.ts`:
   ```typescript
   interface Notification {
     id: string;
     userId: string;
     channel: 'email' | 'sms' | 'push';
     template: string;
     data: Record<string, unknown>;
     status: 'pending' | 'sent' | 'failed';
     createdAt: Date;
   }
   ```

2. Create `notificationQueue.ts` with in-memory queue:
   - `enqueue(notification)` — add to pending queue
   - `dequeue()` — get next pending notification
   - `processAll()` — drain queue with concurrency limit of 10
   - Internal polling loop: check queue every 500ms for new items

3. Create `notificationService.ts` as the main orchestrator:
   - `notify(userId, event, data)` — determine channels, create notifications, enqueue
   - User preference lookup: which channels are enabled per user
   - Rate limiting: max 5 notifications per user per hour
   - Deduplication: don't send same event+user combo twice

### Phase 2: Delivery Channels

1. Create `emailChannel.ts`:
   - SendGrid integration with API key from env
   - HTML template rendering for order emails
   - Delivery tracking via SendGrid webhooks

2. Create `smsChannel.ts`:
   - Twilio integration for SMS delivery
   - Character limit handling and message splitting
   - Delivery status callback URL

3. Create `pushChannel.ts`:
   - Firebase Cloud Messaging for push notifications
   - Device token management per user
   - Badge count tracking

### Phase 3: Event Binding & Templates

1. Create `eventListener.ts` that sets up listeners for order events:
   - On `order.created`: check what happened, build notification payload
   - On `order.shipped`: include tracking number in notification
   - On `order.delivered`: include delivery confirmation
   - On `order.refunded`: include refund amount and timeline
   - Each listener calls `notificationService.notify()` directly

2. Create `templateEngine.ts` with Handlebars templates:
   - `order-confirmation.hbs`, `shipping-update.hbs`, `delivery.hbs`, `refund.hbs`
   - Per-channel variants (HTML for email, plain text for SMS, short for push)

3. Wire up initialization in `app.ts`:
   - Create notification service instance
   - Register event listeners on startup
   - Start queue processing loop

## Verification

1. Unit test notification service: correct channel selection per user preferences
2. Unit test queue: enqueue, dequeue, concurrency limit
3. Unit test each channel: mock SendGrid/Twilio/Firebase, verify payloads
4. Integration test: emit `order.created`, verify notification is queued and sent
5. Test rate limiting: exceed 5 notifications/hour, verify throttling
6. Test deduplication: emit same event twice, verify single notification
