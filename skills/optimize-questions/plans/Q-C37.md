# Plan: Migrate Configuration Format

## Context

Our application currently stores configuration in YAML files. The DevOps team has
standardized on TOML across all services. We need to convert the config format to
align with the org-wide standard. This affects local development configs, CI configs,
and production deployment configs.

## Current State

- Node.js 20, TypeScript 5.3
- Configuration in YAML: `config/app.yaml`, `config/database.yaml`, `config/services.yaml`
- Config loader: `src/config/yamlLoader.ts` using `js-yaml` package
- 15 microservices in the org, 12 already on TOML
- CI/CD pipeline reads config during build step
- Helm charts reference config paths for Kubernetes deployments

## Approach

Convert the config format from the current format to the target format. Update the
config loader to read the new format. Ensure all environments continue to work during
and after the transition.

## Files to Modify

- `src/config/configLoader.ts` (new) — new config loader for target format
- `src/config/yamlLoader.ts` — deprecate, keep as fallback during transition
- `config/app.toml` (new) — converted application config
- `config/database.toml` (new) — converted database config
- `config/services.toml` (new) — converted service discovery config
- `package.json` — add TOML parsing dependency
- `Dockerfile` — update config file references
- `helm/values.yaml` — update config mount paths

## Implementation

### Phase 1: New Loader

1. Add `@iarna/toml` dependency to `package.json`

2. Create `configLoader.ts`:
   - `loadConfig(path: string): AppConfig` — load and parse TOML file
   - Type-safe return using existing `AppConfig` interface
   - Environment variable override support (same as current YAML loader)
   - Validation against JSON Schema

3. Add fallback chain: try TOML first, fall back to YAML if not found
   - Log deprecation warning when YAML fallback triggers
   - Plan to remove YAML fallback in 3 months

### Phase 2: Convert Config Files

1. Convert `config/app.yaml` to `config/app.toml`:
   - Transfer all key-value pairs to TOML equivalent
   - Preserve comments as TOML comments
   - Maintain logical grouping with TOML sections

2. Convert `config/database.yaml` to `config/database.toml`:
   - Connection strings, pool settings, replica configs
   - TOML inline tables for compact nested values

3. Convert `config/services.yaml` to `config/services.toml`:
   - Service URLs, timeouts, retry policies
   - Array of tables for multiple service entries

### Phase 3: Infrastructure Updates

1. Update `Dockerfile`:
   - Copy `.toml` files instead of `.yaml` files
   - Update config path environment variables

2. Update `helm/values.yaml`:
   - Change config file mount paths from `.yaml` to `.toml`
   - Update ConfigMap references

3. Update CI pipeline (`.github/workflows/deploy.yml`):
   - Config validation step uses new TOML schema
   - Build step references new config paths

### Phase 4: Deprecation & Cleanup

1. Mark `yamlLoader.ts` as deprecated with `@deprecated` JSDoc tag
2. Add startup warning: "YAML configuration is deprecated, migrate to TOML"
3. Update developer documentation with new config format examples
4. Schedule YAML loader removal for next major version

## Verification

1. Unit test new TOML loader: parse valid config, reject invalid
2. Unit test fallback chain: TOML present → use it, TOML missing → YAML fallback
3. Unit test environment variable overrides work with TOML loader
4. Integration test: application starts with TOML config, all features work
5. Diff test: compare resolved config from YAML vs TOML — must be identical
6. CI test: pipeline passes with new config paths
