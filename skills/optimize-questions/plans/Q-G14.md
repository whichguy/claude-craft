# Plan: Add Rate Limiting and Request Logging Middleware

## Context

The API has no rate limiting (DoS risk) and no structured request logging (debugging pain).
The ops team needs both before launch. The existing Express API uses conventional middleware:

```
app.use(cors())        // (req, res, next) => { ...; next(); }
app.use(helmet())      // (req, res, next) => { ...; next(); }
app.use(bodyParser.json())
app.use(authMiddleware) // (req, res, next) => { ...; next(); }
app.use('/api/v1', routes)
app.use(errorHandler)  // (err, req, res, next) => { ... }
```

Every middleware follows the `(req, res, next)` pattern, calling `next()` to pass control.

## Approach

Add a rate limiter (sliding window + Redis) and a structured request logger (pino).

## Files to Modify

- `src/middleware/rate-limiter.ts` — Rate limiting middleware
- `src/middleware/request-logger.ts` — Structured request logging
- `src/config/rate-limits.ts` — Per-route rate limit configuration
- `src/app.ts` — Register new middleware
- `src/middleware/rate-limiter.test.ts` — Rate limiter tests
- `src/middleware/request-logger.test.ts` — Logger tests

## Implementation Steps

### Step 1: Create rate limit configuration

Define configs in `src/config/rate-limits.ts`: default (100/min), auth (10/15min),
upload (20/hr), public (200/min). Each has `windowMs`, `maxRequests`, optional `keyGenerator`.

### Step 2: Implement rate limiter middleware

Use Koa-style async handler pattern. Accept a context object with request and state, use
Redis sorted sets for sliding window counting. Return 429 response object when exceeded,
or undefined to continue. Set `X-RateLimit-*` headers. Key by userId or IP.

```ts
export function createRateLimiter(configName: string = 'default') {
  const config = rateLimits[configName] || rateLimits.default;
  return async (ctx: { request: Request; state: Record<string, any> }) => {
    const key = getClientIdentifier(ctx);
    const redisKey = `ratelimit:${configName}:${key}`;
    // Sliding window via sorted set: zremrangebyscore, zadd, zcard, pexpire
    const requestCount = await countInWindow(redisKey, config.windowMs);
    if (requestCount > config.maxRequests) {
      return { status: 429, body: { error: 'Too Many Requests' } };
    }
    return undefined; // continue to next handler
  };
}
```

### Step 3: Implement request logger middleware

Use Koa-style async handler. Assign request ID, create pino child logger on context,
await downstream via `ctx.state.next()`, measure duration, log with method/path/status/
duration/userId/rateLimit info. Log level: error for 5xx, warn for 4xx, info otherwise.

```ts
export async function requestLogger(ctx: { request: Request; state: Record<string, any> }) {
  const start = performance.now();
  ctx.state.requestId = crypto.randomUUID();
  ctx.state.logger = logger.child({ requestId: ctx.state.requestId });
  const response = await ctx.state.next();
  const duration = Math.round(performance.now() - start);
  ctx.state.logger.info({ method, path, status, duration }, 'Request completed');
  return response;
}
```

### Step 4: Register middleware in app.ts

Add `requestLogger` and `createRateLimiter('default')` at top of chain. Add stricter
limiters for auth and upload routes.

### Step 5: Write rate limiter tests

Test: allows under limit, returns 429 when exceeded, resets after window, tracks by user ID
when authenticated, tracks by IP for anonymous, applies per-route config.

### Step 6: Write request logger tests

Test: logs with timing info, assigns unique request ID, error level for 5xx, warn for 4xx,
includes rate limit info, creates child logger.

## Risks

- Redis dependency for rate limiting; need graceful fallback
- Sliding window sorted sets use more memory than fixed windows
- Log volume at high traffic needs aggregation infrastructure
